## 客户叙事（Customer Narrative） {#sec-u5ba2huu53d9u4e8bcustomer-narrat}
**When** 一个独立开发者或超级个体需要管理一个完整的软件项目，从需求脑暴到设计、编码、测试、上线，**I want to** 在一个本地平台内看到项目的全生命周期状态、一键触发 AI Skill 执行、在关键节点快速确认风险和产物，**so I can** 不再在多个工具（IDE、文档、命令行、看板）之间切换，确保每一步都留痕、可追溯、关键决策不遗漏。

## 核心问题（Core Problem） {#sec-hexinwenticore-problem}
当前超级个体使用 AI 辅助开发时，面临系统性"过程黑盒"问题：AI 执行了哪些 Skill、生成了什么产物、卡在哪个 Gate、是否需要人工干预，这些信息散落在命令行输出和文件系统中，无法一眼看清全局。同时，随着项目复杂度不同（简单脚本 vs 微服务架构），缺乏按规模自动调整执行策略的机制。本平台通过内置 GTPlanner V2（复杂度路由 + C4 架构穿透 + 架构漂移检测）+ Skill Flow 编排引擎 + 可视化驾驶舱，将 AI 辅助开发的完整 SDLC 过程显性化、可干预、可适配。

## 边界范围（Scope） {#sec-u8fb9u754cfanweiscope}
### In Scope {#sec-in-scope}
- SDLC 全流程可视化：需求探索、概要需求、详细需求、概要设计、详细设计、接口契约、计划拆解、编码实现、单元测试、集成测试、UAT、发布管理、归档收尾、监控分析。
- Draft/Active 双态模型支持：预立项 Draft 态（轻量分析）与正式执行 Active 态（完整流程）。
- 四道 Gate 自检确认：Gate-1（概要需求）、Gate-2.5（详细需求）、Gate-2（概要设计）、Gate-3（UAT），面向超级个体的"AI 总结 + 用户快速确认"模式。
- HITL Waiting 状态：Skill 执行到 Gate 节点时暂停，等待用户确认后继续。
- Skill Flow 编排引擎：YAML 驱动 DAG 调度、并行执行、条件分支、失败重试、产物传递。
- 可视化画布：拓扑图（节点+连线）、泳道视图、列表视图，支持缩放/拖拽/筛选。
- 阶段详情面板：Skill 指令快照、输入/输出产物、执行日志、质量门禁结果。
- 产物浏览器：Markdown/Mermaid 渲染、OpenAPI/Swagger 预览、测试报告可视化。
- 审批中心：待确认队列、历史决策追溯。
- **GTPlanner V2 内置实现**：复杂度路由面板（Simple/Standard/Complex）、C4 架构穿透浏览、架构漂移检测，不依赖外部 Docker。
- **管理员预制模板驱动**：平台内置不同规模项目模板，模板定义阶段与 Skill 绑定关系；用户选择模板后按模板匹配实际导入的 Skill。
- **多平台 Adapter 接口预留**：设计层抽象 AI 平台 Adapter 接口，MVP 仅实现 Kimi CLI Adapter。
- **双向同步**：平台内编辑产物后可写回 `openspec/changes/{变更名}/` 目录；文件系统优先冲突解决策略。
- Kimi CLI 单一平台执行（MVP）：本地 CLI 调用、stdout/stderr 捕获、产物目录监听。
- 用户手动注册/导入 Skill 路径：平台动态解析 SKILL.md Frontmatter 生成画布节点。
- 本地单机部署：零运维，SQLite 存储元数据，产物存本地文件系统，10 项目上限（MVP）。

### Out of Scope {#sec-out-of-scope}
- 多租户与企业级 RBAC（P1 后考虑）。
- 多 AI 平台适配实现（Claude/Cursor/MCP），MVP 仅接口预留 + Kimi CLI 实现。
- 通用项目管理（如替代 Jira 的完整需求管理）。
- 外部 CI/CD 工具链替代（如替代 GitHub Actions）。
- 底层 LLM 基座选型与训练。
- 产物云端存储/SaaS 化（MVP 仅本地文件系统）。
- 实时协同编辑（多用户同时操作同一项目）。

### 兼容性要求 {#sec-jianrongxingyaoqiu}
- 需兼容 Arsitect 项目的 Skill 目录结构（SKILL.md + meta.json）。
- 产物目录结构需兼容 `openspec/changes/{change}/` 规范。
- 支持主流浏览器：Chrome / Edge / Firefox / Safari 最新 2 个主版本。

## 业务假设（Business Assumptions） {#sec-yewuu5047shebusiness-assumptions}
1. 超级个体在使用 AI 辅助开发时，最大的痛点不是"AI 能力不够"，而是"过程不可见、产物难管理、关键节点容易遗漏"。
2. 用户愿意为"过程可视化"接受一定的本地环境配置成本（安装 Kimi CLI、注册 Skill 路径）。
3. 平台的核心差异化不是替代现有工具（Jira/Notion），而是将 AI 执行过程本身可视化，这是传统项目管理工具无法覆盖的。
4. 单人场景下的"审批"本质上是"自检确认"，UI 上应避免传统审批流的沉重感，强调"快速确认+风险提醒"。
5. 项目规模差异（简单脚本 vs 微服务）需要差异化的执行策略，模板驱动的复杂度路由能降低用户的决策负担。

| 模块 | 职责 | 优先级 | 备注 |
|------|------|--------|------|

| 前端面板层（Stage Detail + Artifact Viewer） | 阶段详情抽屉、产物浏览器（Markdown/Mermaid/Swagger/测试报告）、产物编辑（支持双向同步写回） | P0 | 右侧滑出抽屉，多模态渲染 + 文本编辑 |
| 前端工作台（Project Dashboard） | 项目列表、健康度卡片、团队效率统计、风险预警 | P0 | 首屏入口 |
| 前端审批中心（Gate Center） | 待确认队列、自检摘要、快速确认、历史追溯 | P0 | 单人场景下强调"总结+确认" |
| 前端模板中心（Template Center） | 管理员预制模板管理、模板选择向导、阶段-Skill 绑定可视化 | P0 | GTPlanner V2 核心入口 |
| 后端项目管理服务 | Workspace/Application/Project/Module CRUD、健康度计算 | P0 | FastAPI + SQLite |
| 后端流程编排引擎 | YAML 解析、DAG 构建、拓扑排序、并行调度、条件评估、超时监控 | P0 | Skill Flow Runtime 核心 |
| 后端 Skill 调度服务 | Skill Adapter（Kimi CLI）、输入注入、输出捕获、日志收集、重试；预留多平台 Adapter 接口 | P0 | 进程级调用 + 产物目录监听 |
| 后端 Gate 服务 | Waiting 状态管理、自检摘要生成、确认/驳回/重试、超时处理 | P0 | 单人模式下摘要由 AI 生成或模板生成 |

| 后端 C4 架构服务 | C4 DSL 解析、架构图生成、层级穿透浏览、架构快照存储 | P1 | 内置 DSL 解析器和渲染器 |
| 后端架构漂移检测服务 | 对比历史架构快照与当前设计，识别未登记的架构变更 | P1 | 后台异步任务 |

## 风险点（Risks） {#sec-fengxianu70b9risks}
| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| GTPlanner V2 内置实现（C4 DSL 解析 + 复杂度路由 + 漂移检测）工作量超预期，挤压 MVP 核心可视化功能 | 高 | MVP 延期或核心体验不完整 | C4 渲染降级为调用 Structurizr 库；复杂度路由改为基于规则引擎（文件数阈值）而非 AI 深度分析；漂移检测 P1 后实现 |
| Kimi CLI 的进程级执行模型难以提供实时中间状态，导致画布"假死"感 | 高 | 用户体验差，无法感知 Skill 执行进度 | 设计"已触发/运行中/产物生成中"三级伪状态；CLI 执行前显示预期耗时；产物目录实时监听增量更新 |
| SKILL.md 自动解析上下游 Skill 引用准确率不足，导致 DAG 构建错误 | 中 | 画布节点依赖关系错乱，执行顺序错误 | 前置校验 + 容错渲染（解析失败时标记为"需手动配置"）；提供可视化 DAG 编辑器供用户手动修正 |
| SQLite 在频繁写入执行日志和状态快照时可能成为瓶颈 | 中 | 状态更新延迟、UI 卡顿 | 日志批量写入、状态变更 debounce、预留 PostgreSQL 迁移路径 |
| Skill YAML 格式不统一或 Frontmatter 解析失败导致节点无法渲染 | 中 | 画布空白或节点缺失 | 前置校验 + 容错渲染（缺失字段显示默认值 + 警告提示） |
| 超级个体对"四道 Gate"产生流程负担，反而降低效率 | 中 | 用户流失、产品口碑下降 | Gate UI 极简设计（一键确认+风险提示折叠），允许在设置中关闭非关键 Gate（需显式声明风险自负） |
| 产物文件与数据库状态不一致（如外部修改了产物文件） | 低 | 详情面板显示过期内容 | 文件哈希校验 + 手动刷新按钮 + 文件系统事件监听；已采用文件系统优先策略 |
| 双向同步时用户误操作覆盖外部重要修改 | 低 | 数据丢失风险 | 平台保存前检测外部变更并弹窗提示；保留产物版本快照（基于 Git 或本地备份） |

## 待确认项（Open Questions） {#sec-daiquerenu9879open-questions}
1. C4 DSL 语法规范：内置 C4 解析器采用哪种 DSL 格式（Structurizr DSL、Mermaid C4 扩展、还是自定义 YAML）？→ 建议由 Tech Lead 在概要设计前确认。
2. Kimi CLI "实时联动"的具体技术协议：是平台调用 CLI 后轮询产物目录，还是 CLI 支持某种进度输出格式（JSON Lines）？→ 建议由 Tech Lead 在概要设计前做快速 POC 验证。
3. Gate 自检摘要的生成方式：由平台根据产物规则硬编码判断，还是调用 AI（Kimi CLI）生成自然语言摘要？→ 建议由 PM + Tech Lead 在详细设计前确认。

## 对抗性自我批判（Adversarial Critique） {#sec-duikangxingziu6211piu5224adversa}
> 强制列出至少 3 个产品/方案的弱点，以及缓解措施。

1. **弱点**：GTPlanner V2 内置实现（C4 DSL 解析 + 复杂度路由 + 漂移检测）工作量巨大，可能挤压核心可视化功能的开发周期，导致 MVP 成为一个"半生不熟"的架构工具而非可视化驾驶舱。
   **缓解**：设定硬性范围边界——C4 渲染首期仅支持 L1/L2 上下文图和容器图，不实现 L3 组件图；复杂度路由基于简单的文件数/实体数阈值规则，不引入 AI 深度分析；架构漂移检测 P1 后实现。若内置工作量仍超预期，C4 降级为调用 Structurizr 开源库。

2. **弱点**：Kimi CLI 的进程执行模型天然不适合"实时可视化"，用户可能在 Skill 执行期间看到长时间无反馈的黑盒状态，体验甚至不如命令行直接观察输出。
   **缓解**：设计"预期进度条"（基于历史同类 Skill 耗时估算）+ 产物增量监听（文件一旦生成立即展示）+ CLI stdout 实时流式捕获（如果 Kimi CLI 支持）。若不支持流式，则提供"查看实时日志"按钮打开终端面板。

3. **弱点**："管理员预制模板"模式在单人使用场景下显得多余——超级个体通常没有"管理员"角色，自己就是管理员和使用者。模板维护可能成为用户的额外负担，且 Simple/Standard/Complex 的边界模糊，用户可能频繁选错。
   **缓解**：模板设计为"智能推荐 + 一键应用"，而非强制选择。平台根据用户初始输入（项目描述 + 预估规模）自动推荐模板，用户可一键接受或手动调整。允许用户保存自定义模板为个人默认值，减少重复选择。

## 技术线索（Technical Hints） {#sec-u6280u672fxiansuotechnical-hints}
- 推荐技术方向：
  - 前端画布：React-Flow (@xyflow/react) 支持动态节点、分组、泳道、自定义节点渲染，与需求完全匹配。
  - 后端编排：可参考 Prefect / Dagster 的轻量级 DAG 执行器设计，但需简化至单进程内调度。
  - CLI 集成：Python `subprocess` + `asyncio` 异步执行，配合 `watchdog` 库监听产物目录变更。
  - 状态持久化：SQLite 足够支撑 MVP；使用 `aiosqlite` 实现异步操作。
  - 实时通信：前端-后端使用 WebSocket（python-socketio / Socket.io-client），推送 Skill 状态变更和 Gate 等待通知。
  - C4 渲染：首期评估 Structurizr 的 Python 绑定或将其 DSL 解析器嵌入；若内置工作量过大，降级为前端嵌入 Structurizr 的 Web 组件。
  - 复杂度路由：基于规则引擎（文件数阈值、实体数统计），暂不引入 LLM 深度分析。

- 需避免的技术债务：
  - 不要将 Skill 节点写死在前端代码中，必须完全动态化（基于用户导入的 YAML/Frontmatter 渲染）。
  - 不要在 MVP 引入 ORM 过度设计，SQLAlchemy 2.0 的 Mapped[] 注解已足够，避免过早抽象 Repository 层。
  - 不要试图在 MVP 实现真正的分布式执行或多 Worker，单进程异步足够。
  - GTPlanner V2 的内置实现需设定硬性边界，防止无限膨胀（首期仅实现模板匹配 + C4 L1/L2 图 + 基于规则的复杂度路由）。

- 性能预期：
  - 首屏加载 < 2s（Lighthouse 性能分 >= 90）。
  - 典型节点数拓扑图交互帧率 >= 60fps（React-Flow 默认优化）。
  - Skill 状态变更端到端延迟 < 5s（轮询模式下可接受）。
  - 产物渲染 < 500ms（Markdown 分页加载）。

### In Scope {#sec-in-scope_1}
- SDLC 全流程可视化：需求探索 -> 概要需求 -> 详细需求 -> 概要设计 -> 详细设计 -> 接口契约 -> 计划拆解 -> 编码实现 -> 单元测试 -> 集成测试 -> UAT -> 发布管理 -> 归档收尾 -> 监控分析。

- 四道 Gate 自检确认：Gate-1（概要需求）、Gate-2.5（详细需求）、Gate-2（概要设计）、Gate-3（UAT），面向超级个体的"AI 总结 + 用户快速确认"模式。
- HITL Waiting 状态：Skill 执行到 Gate 节点时暂停，等待用户确认后继续。
- Skill Flow 编排引擎：YAML 驱动 DAG 调度、并行执行、条件分支、失败重试、产物传递。
- 可视化画布：拓扑图（节点+连线）、泳道视图、列表视图，支持缩放/拖拽/筛选。
- 阶段详情面板：Skill 指令快照、输入/输出产物、执行日志、质量门禁结果。
- 产物浏览器：Markdown/Mermaid 渲染、OpenAPI/Swagger 预览、测试报告可视化。
- 审批中心：待确认队列、历史决策追溯。
- Kimi CLI 单一平台适配：本地 CLI 调用、stdout/stderr 捕获、产物目录监听。
- 用户手动注册/导入 Skill 路径：平台动态解析 SKILL.md Frontmatter 生成画布节点。
- 本地单机部署：零运维，SQLite 存储，10 Project 上限（MVP）。

1. **弱点**：超级个体市场付费意愿低，且本地单机部署难以形成 SaaS 订阅收入。平台若始终面向单人，商业可持续性存疑。

| Q2: 与 Jira/Notion/看板工具相比，优势是什么？ | 传统工具管理的是"人完成的任务"，本平台管理的是"AI 执行的 Skill"。它能展示 AI 的实时执行状态、自动生成的产物（设计文档/接口契约/测试报告）、以及 AI 产出的质量门禁结果。这是传统项目管理工具无法做到的。 |
| Q3: 我需要付费订阅 Kimi 吗？ | 平台本身不绑定特定 LLM 服务。MVP 阶段通过本地 Kimi CLI 执行，你的 Kimi CLI 配置和账户由你自己管理。未来会支持更多 AI 平台。 |
| Q4: 我的代码和数据安全吗？ | 平台完全本地运行（SQLite + 本地文件系统），产物和项目数据不会上传到云端。除非你主动配置远程备份，否则所有数据都在你的机器上。 |
| Q5: 我是一个人开发，为什么还需要 Gate 审批？ | 这里的 Gate 不是"等领导签字"，而是"AI 帮你做自检摘要，你快速确认风险点"。它帮你避免"漏看设计文档就直接编码"或"漏写回滚方案就上线"的低级错误。 |

| 问题 | 答案 |
|------|------|
| Q1: MVP 是否可交付？技术可行性如何？ | 高。核心技术栈（React 19 + FastAPI + SQLite）成熟；React-Flow 已被验证可支撑复杂节点图；Kimi CLI 本地调用为简单进程管理。最大风险在于 CLI 实时状态捕获，可通过轮询+产物监听缓解。 |
| Q2: 是否需要修改现有 Arsitect 项目的代码？ | 否。本平台是独立产品，通过读取/调用 Arsitect 的 Skill 定义和 CLI 与之集成，不修改 Arsitect 本身。 |
| Q3: 产物存储用数据库还是文件系统？ | 待定（见待确认项 Q1）。倾向方案：执行状态和元数据存 SQLite，产物文件（Markdown/YAML）存本地文件系统（兼容 openspec 目录结构），平台通过路径关联两者。 |
| Q4: 单人 Gate 体验如何与传统多人审批区分？ | 交互层取消"签字人/会签"概念，改为"阶段总结卡片"：AI/模板自动生成关键风险点和待补充项清单，用户一键"确认通过"或"返回补充"。历史记录仍保留时间戳和评语字段，为未来协作预留。 |
| Q5: 平台是否需要在后台常驻运行？ | MVP 阶段为"启动即用"的本地 Web 服务（前端静态 + Python 后端双进程），用户启动后通过浏览器访问，关闭即停止。无需系统级守护进程。 |

- 推荐技术方向：
  - 前端画布：React-Flow (@xyflow/react) 支持动态节点、分组、泳道、自定义节点渲染，与需求完全匹配。
  - 后端编排：可参考 Prefect / Dagster 的轻量级 DAG 执行器设计，但需简化至单进程内调度。
  - CLI 集成：Python `subprocess` + `asyncio` 异步执行，配合 `watchdog` 库监听产物目录变更。
  - 状态持久化：SQLite 足够支撑 MVP；使用 `aiosqlite` 实现异步操作。
  - 实时通信：前端-后端使用 WebSocket（python-socketio / Socket.io-client），推送 Skill 状态变更和 Gate 等待通知。

- 需避免的技术债务：
  - 不要将 Skill 节点写死在前端代码中，必须完全动态化（基于用户导入的 YAML/Frontmatter 渲染）。
  - 不要在 MVP 引入 ORM 过度设计，SQLAlchemy 2.0 的 Mapped[] 注解已足够，避免过早抽象 Repository 层。
  - 不要试图在 MVP 实现真正的分布式执行或多 Worker，单进程异步足够。

## 外部 FAQ {#sec-u5916bu-faq}
**Q: 这次升级会改变我现有的 Skill 使用方式吗？**
A: 不会。41 个 Skill 的触发条件和行为规范保持不变。升级主要在底层编排和架构引擎，对用户交互层透明。

**Q: 我需要安装新的软件吗？**
A: 基础功能无需安装。若需要架构自动渲染或 OpenHands 沙箱执行，需可选安装 Docker 和 C4 InterFlow CLI。

**Q: 架构漂移检测会阻断我的发布吗？**
A: 不会直接阻断。漂移报告作为 Gate 2 评审的输入，由 Tech Lead 决定是否接受偏差。

## 内部 FAQ {#sec-u5185bu-faq}
**Q: 为什么不直接 fork C4 InterFlow 而是 CLI 调用？**
A: C4 InterFlow 是 .NET 生态，arsitect 核心是 Python/Node 技术栈。Fork 后跨语言维护成本极高。MIT 协议允许直接调用发布产物。

**Q: Coco Workflow 为什么不直接集成？**
A: Coco 是 Claude Code 专属插件。直接引入会绑定到 Claude Code 生态，丧失平台中立性（Kimi/Cursor/Codex 用户无法使用）。

**Q: OpenHands 为什么不嵌入源码？**
A: OpenHands 依赖树庞大（70K stars 项目），嵌入会导致部署包膨胀 10 倍以上，且破坏 Docker 安全沙箱边界。
