| 建议角色 | 职责 | 必须出席 | 评审重点 |

| **PM（产品经理）** | 确认需求完整性、业务规则闭环、竞品差异策略 | 是 | 客户叙事、数据口径、假设登记册、非目标 |
| **Tech Lead** | 确认技术可行性、架构决策合理性、风险缓解措施 | 是 | 模块初分、技术线索、GTPlanner V2 内置边界、AI 原语选型 |

| **Arsitect 生态负责人**（如有） | 确认与现有 Skill 规范的兼容性、双向同步策略 | 建议 | Skill 目录结构兼容、openspec 目录规范、DAG 解析准确性 |

## 2. 静默阅读指南（15 分钟）

### 阅读顺序与批注建议

```text
第 1-3 分钟：执行摘要 + 客户叙事
  → 批注：客户痛点是否真实？JTBD 格式是否完整？

第 4-6 分钟：边界范围（In Scope / Out of Scope）
  → 批注：GTPlanner V2 内置范围是否过大？C4 穿透/漂移检测是否应 P1？
  → 批注：多平台 Adapter 预留接口是否增加不必要的前期复杂度？

第 7-9 分钟：数据口径声明 + 假设登记册
  → 批注：所有关键数值是否已锁定？冲突是否已显式处理？
  → 批注：假设 A-008（GTPlanner 工作量）置信度是否应降级为"低"？

第 10-12 分钟：模块初分 + 风险点
  → 批注：模块优先级划分是否合理？P0 是否过多？
  → 批注：风险缓解措施是否足够具体，可执行？

第 13-15 分钟：对抗性自我批判 + 待确认项
  → 批注：3 个弱点是否击中要害？是否有遗漏的致命弱点？
  → 批注：待确认项的责任人和时间节点是否明确？
```

### 阅读批注标记规范

| 标记 | 含义 | 示例 |
|------|------|------|
| **[BLOCKER]** | 阻塞项，必须通过才能进入下游 | "GTPlanner V2 内置工作量未细化到人天，无法评估可行性" |
| **[QUESTION]** | 疑问项，需会议讨论澄清 | "模板匹配时若用户导入的 Skill 不足，是否允许执行？" |
| **[SUGGESTION]** | 建议项，非阻塞 | "建议将 C4 L3 组件图明确排除在 P0 外" |
| **[PRAISE]** | 认可项 | "文件系统优先的冲突策略符合开发者习惯" |

## 3. 问答议程（40 分钟）

### 第一阶段：客户叙事与价值验证（8 分钟）

| 议题 | 提问人 | 目标 |
|------|--------|------|
| JTBD 完整性 | PM | 确认 "When / I want to / so I can" 是否覆盖核心场景 |
| 差异化验证 | PM | 确认与 Dify/CrewAI/Jira 的差异化策略是否成立 |
| 目标用户聚焦 | Designer | 确认"超级个体"定义是否过于狭窄，是否遗漏小团队场景 |

**预期冲突点**：
- PM 可能认为 GTPlanner V2 的模板驱动模式对超级个体过于"重"，建议改为纯动态。
- 应对：模板设计为"智能推荐 + 一键应用"，非强制选择；用户可保存个人默认模板。

### 第二阶段：假设与数据口径（8 分钟）

| 议题 | 提问人 | 目标 |
|------|--------|------|
| 数据口径锁定 | PM | 确认 10 项关键数值无跨文件矛盾 |
| 假设置信度 | Tech Lead | 评估 A-008（GTPlanner 工作量）和 A-009（文件系统优先）的推翻条件 |
| 模板覆盖度 | Tech Lead | 确认 Simple/Standard/Complex 模板是否能覆盖 90% 场景 |

**预期冲突点**：
- Tech Lead 可能质疑 GTPlanner V2 内置实现的工期，要求提供人天估算或范围裁剪方案。
- 应对：已准备三级降级方案（Structurizr 库调用、规则引擎替代 AI、P1 后实现漂移检测）。

### 第三阶段：风险与缓解措施（10 分钟）

| 议题 | 提问人 | 目标 |
|------|--------|------|
| 最高风险排序 | Tech Lead | 确认"GTPlanner 工作量挤压"和"CLI 实时状态"两个高风险的优先级 |
| 缓解措施可执行性 | PM + Tech Lead | 每项高风险的缓解措施是否已落实到具体人 |
| 回滚机制 | Tech Lead | 确认 AI 错误触发回滚的条件和实现机制是否可行 |

**预期冲突点**：
- Tech Lead 可能认为"文件系统优先"策略在产品场景下不合理，建议改为乐观锁。
- 应对：展示历史调研结论（开发者更信任外部编辑器），同时承诺若用户反馈超过 30% 则改为乐观锁。

### 第四阶段：模块与架构（10 分钟）

| 议题 | 提问人 | 目标 |
|------|--------|------|
| P0 模块数量 | Tech Lead | 评估 12 个 P0/P1 模块在 6-8 周内的可交付性 |
| AI 原语选型 | Tech Lead | 确认 Agent/Subagent 仅用于 Skill 执行，其余模块用 Skill 是否合理 |
| 多平台 Adapter 预留 | Tech Lead | 评估接口预留是否增加过度设计，是否应推迟至 P1 |
| 双向同步实现 | Tech Lead | 确认文件系统优先策略的技术实现方案（watchdog + 哈希校验） |

**预期冲突点**：
- Tech Lead 可能认为多平台 Adapter 预留是过早抽象，建议 MVP 硬编码 Kimi CLI。
- 应对：Adapter 接口仅为抽象层（Interface + 一个实现类），不增加实际工作量；删除预留反而增加 P1 重构成本。

### 第五阶段：冻结决策（4 分钟）

| 议题 | 决策人 | 目标 |
|------|--------|------|
| 需求冻结签字 | PM + Tech Lead | 双方书面确认 PRD-000 可进入详细需求阶段 |
| 待确认项分配 | PM | 为 5 个待确认项指定责任人和截止日期 |
| 下游文档影响 | PM | 明确哪些下游文档需同步更新（如 UI 规格书需从 25 节点改为动态节点） |

## 4. 评审前自检清单（评审者用）

在会议开始前，请每位评审者独立完成以下检查：

- [ ] 我已完整阅读 requirement-draft.md（预计 15 分钟）
- [ ] 我已阅读 ai-architecture-decision.md（预计 8 分钟）
- [ ] 我已在文档中添加至少 3 条批注（含标记类型）
- [ ] 我理解本次变更的核心目标是"AI 辅助开发过程可视化"而非"通用项目管理"
- [ ] 我确认本变更不涉及修改现有 Arsitect 仓库代码
- [ ] 我已准备好就 GTPlanner V2 内置范围进行讨论

## 5. 评审后产出物

评审会议结束后，需产出以下文档：

1. **评审会议纪要**：记录所有 [BLOCKER]、[QUESTION] 的讨论结论
2. **需求变更申请单**：若有范围调整，更新 requirement-draft.md 版本号并标注影响范围
3. **冻结签字页**：PM 和 Tech Lead 双方书面确认（可附在本文档末尾）
4. **待确认项跟踪表**：为 5 个待确认项指定责任人和截止日期

## 评审者清单

- PM（产品负责人）：确认客户叙事和业务规则，尤其是超级个体场景下的 Gate 体验重塑。
- Tech Lead：确认技术可行性和集成依赖，尤其是 Kimi CLI 实时联动方案和产物存储策略。
- Designer：确认交互和视觉范围，尤其是 Flow Canvas 的动态节点渲染和单人 Gate 确认流程。

## 静默阅读指南

- 时长：15 分钟
- 材料：requirement-draft.md、research-report.md
- 任务：阅读并批注疑问，不展开讨论。重点关注：
  1. 单人 Gate 自检确认模式是否符合超级个体的心理模型。
  2. 动态 Skill 导入机制是否会导致画布布局混乱。
  3. Kimi CLI 进程级调用与实时状态推送的架构矛盾是否有未考虑到的缓解方案。

## 问答议程（40 分钟）

### 第一部分：客户叙事与范围（10 分钟）
- Q1：客户叙事是否准确反映超级个体的痛点？"过程黑盒"是否是最痛的点？
- Q2：In Scope / Out of Scope 边界是否清晰？MVP 是否可交付？
- Q3：单人 Gate 确认模式是否会让用户觉得"多此一举"？如何平衡"不遗漏关键节点"与"不增加流程负担"？

### 第二部分：假设与数据口径（10 分钟）
- Q4：假设登记册中的 A-004（CLI 实时状态获取）是否需要在设计阶段前做快速 POC 验证？
- Q5：数据口径声明中的关键数值是否存在冲突口径？是否已显式声明冲突及采用理由？
- Q6：数据口径声明中的"确认状态"是否为"锁定"？若为"预计"或"待验证"，是否已列入待确认项并分配责任人？

### 第三部分：风险与模块（20 分钟）
- Q7：风险等级和缓解措施是否可接受？"Kimi CLI 实时状态延迟"风险是否有更优解法？
- Q8：模块初分的优先级是否合理？MVP 是否可交付？
- Q9：产物存储策略（数据库 vs 文件系统）是否需要在评审会上拍板？
- Q10：平台与 Arsitect 仓库的兼容深度（单向读取 vs 双向同步）是否需要在 PRD 阶段明确？

## 静默阅读指南（15 分钟）

请按以下顺序阅读，并在文档右侧批注：

1. **requirement-draft.md**（5 分钟）
   - 重点：客户叙事是否真实反映痛点？
   - 批注：假设登记册中哪些假设置信度过低？

2. **research-report.md**（3 分钟）
   - 重点：六个开源项目的复用策略是否合理？
   - 批注：是否有遗漏的备选方案？

3. **ai-architecture-decision.md**（4 分钟）
   - 重点：Agent/Skill 的选型是否符合实际复杂度？
   - 批注：上下文管理策略是否过于乐观？

4. **GTPlanner.txt（原始素材）**（3 分钟）
   - 重点：调研结论与推荐方案是否一致？
   - 批注：实施优先级是否合理？

### 第一部分：客户叙事与需求边界（10 分钟）

1. 客户叙事中的两个场景（bug fix 快速通道 + 新微服务全量流程）是否覆盖了 80% 的使用情况？
2. "不在范围"的四项是否过于保守？是否有应该纳入的内容？
3. 复杂度四级的阈值（1 文件 / 3 文件 / 跨服务）是否需要数据支撑？

### 第二部分：技术选型与风险（15 分钟）

4. C4 InterFlow 作为唯一 AaC 底座，若社区停滞，备选方案（Structurizr / 自研 DSL）何时启动评估？
5. OpenHands Docker 镜像的体积和启动时间是否已实测？若未实测，W3 的接入计划是否需要调整？
6. 多执行器状态一致性方案（Git commit 为边界）是否足够？是否需要引入事务性提交？
7. 架构漂移检测的误报容忍度（< 15%）如何量化？是否有基线数据？

### 第三部分：AI 架构与工程纪律（10 分钟）

8. 四个模块被评为 Agent/Subagent，当前平台的 Subagent 调度能力是否足够？
9. 上下文压缩规则（80K 触发摘要）是否经过验证？对 C4 模型这种结构化数据是否有效？
10. AI 模型错误触发回滚的 4 个条件中，哪些可以自动化？哪些必须人工确认？

### 第四部分：实施计划与资源（5 分钟）

11. 6 周实施计划中，W1-W2 的关键路径是否存在并行空间？
12. 本次变更是否需要新增团队成员（如 .NET 工程师维护 C4 InterFlow 集成）？

## 评审通过标准

- [ ] 客户叙事获得 PM 确认
- [ ] Tech Lead 确认外部依赖风险可控
- [ ] Architect 确认 DSL 扩展设计可行
- [ ] DevOps 确认 Docker 部署方案可落地
- [ ] 至少 3 个对抗性弱点已讨论并达成共识

## 评审后动作

- **通过**：进入 `prd-generation` 阶段，输出 PRD-000 五文件概要需求。
- **有条件通过**：记录修改项，修改后无需二次评审。
- **驳回**：返回 brainstorming，补充调研或调整方案。
