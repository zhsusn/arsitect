调研了一圈，2026 年开源生态里出现了几个与 arsitect 定位高度同频的项目。结论是：GTPlanner 借鉴架构、OpenUI 依赖服务、Wireframe 借鉴模式自研，同时必须引入 C4 InterFlow 作为架构基础设施。
一、调研补充：三个不可忽视的开源方案
1. C4 InterFlow —— 架构即代码的基础设施（必看）
这是本次调研最关键的补充。C4 InterFlow 是一个开源的 Architecture as Code (AaC) 框架，直接支持：
DSL 描述架构：用 C# / YAML / JSON 定义 C4 模型的 Structure + Interface + Flow
从代码反向生成：扫描 C# 源码自动生成 Architecture as Code（可扩展到其他语言）
CLI 批量出图：一条命令生成 C4 Context / Container / Component 图 + 序列图 + 文档
CI/CD 原生：GitHub Actions 里自动渲染架构图并发布到 GitHub Pages
架构查询：JSONPath-like 语法查询依赖关系，做影响分析
它与 arsitect 的契合度极高——arsitect 追求的"架构设计+代码生成"闭环，C4 InterFlow 已经解决了架构模型与代码之间的鸿沟问题。
2. Coco Workflow —— 自适应复杂度路由（值得借鉴）
Coco Workflow 是一个面向 Claude Code 的自主 spec-driven 开发插件，其自适应复杂度路由机制非常精妙：
表格
复杂度层级	触发信号	执行路径
Trivial	单文件、bug fix	hotfix — 无 epic，无仪式
Light	1-3 文件、单故事	Design + import — 跳过任务分解
Standard	多文件、有依赖	完整流水线：interview → design → tasks → import
arsitect 完全可以借鉴这个思想：简单需求走"快速架构草图"，复杂需求走"完整 C4 + 接口契约 + 原型验证"。
3. OpenHands —— 全自动代码执行器（可选依赖）
OpenHands（原 OpenDevin）采用 CodeAct 架构，将 Bash、Python、Browser 统一为动作空间，在 Docker 沙箱中自主执行。SWE-bench 验证性能达到 77%。
arsitect 不需要自己实现代码生成，但可以把 OpenHands 作为可选执行器：Context Package 输出后，直接喂给 OpenHands 做全自动施工。
二、三工具策略决策：借鉴 vs 融合 vs 依赖
表格
工具	策略	决策理由	arsitect 中的角色
GTPlanner	借鉴（架构思想）	其 PocketFlow 节点生命周期（prep-exec-post）和 StageGate 机制是通用编排范式，但 GTPlanner 是通用规划工具（写 PRD、做调研），没有 C4 模型、接口契约、领域驱动等软件架构专用节点。直接融合会导致节点语义不匹配。	提取编排内核，自研 ArchitectNode 体系
OpenUI	依赖服务（Docker + API）	OpenUI 是成熟的前端原型渲染引擎，多框架（React/Vue/Svelte）、实时预览、Docker 一键部署。arsitect 不需要自己实现"自然语言 → 前端代码"的复杂管线，直接调用 HTTP API 即可。	作为 PrototypeGenerator 的渲染后端
AI Wireframe Generator	借鉴（LangGraph 模式）+ 自研	其"需求解析→布局规划→SVG 渲染"的 LangGraph 多 Agent 流水线值得借鉴，但通用 Wireframe 无法自动将 C4 Container 映射到页面（如"RiskControl Service"对应"风控仪表盘"）。arsitect 需要领域感知的线框引擎。	自研 WireframeEngine，参考其 Agent 编排模式
三、为什么 GTPlanner 不能融合或依赖？
GTPlanner 的设计目标是通用项目规划（任何行业、任何项目类型），而 arsitect 是软件架构专用编排器。两者在节点语义上存在根本差异：
表格
维度	GTPlanner 节点	arsitect 需要的节点
规划对象	通用任务、里程碑、资源	C4 Context/Container/Component、领域实体、聚合根
技术栈推荐	基于向量检索匹配通用工具	基于架构特征匹配框架（如"微服务+事件驱动"→Kafka+Spring Cloud）
输出格式	Markdown PRD、调研报告	PlantUML C4 图、OpenAPI YAML、DDL SQL、架构决策记录 ADR
反馈循环	用户说"这里不对"	原型点击发现接口缺失，自动回退到 ContractDesigner
如果直接依赖 GTPlanner：你需要在它的通用节点上套一层"软件架构翻译器"，复杂度反而更高。不如借鉴其PocketFlow 三阶段生命周期 + Shared Store + Action 驱动流转的架构思想，自己实现 arsitect-orchestrator。
四、为什么 OpenUI 应该依赖服务？
OpenUI 的定位是前端原型生成专家，它的核心能力（自然语言 → 多框架组件代码 → 实时预览）与 arsitect 的架构设计主线正交：
arsitect 不需要理解 React/Vue 的组件化细节
arsitect 不需要维护前端代码生成模型
arsitect 只需要输入：页面描述 + 组件清单 + 数据流约束；输出：可交互 HTML 原型
集成方式：
yaml
# arsitect/config.yaml
services:
  openui:
    endpoint: "http://openui:7878/api/generate"
    timeout: 120
    output_formats: ["html", "react"]
    max_pages_per_batch: 5
arsitect 的 PrototypeGenerator 将 C4 Container 图和接口契约转换为 OpenUI 可理解的提示词，通过 HTTP 调用获取原型代码。这是成本最低、效果最好的策略。
五、为什么 Wireframe 要借鉴+自研？
AI Wireframe Generator 的LangGraph 多 Agent 架构（需求解析 Agent → 布局规划 Agent → SVG 渲染 Agent）是优秀的工程实践，值得借鉴。但直接依赖它有两个致命问题：
无领域感知：它不知道"RiskControl Bounded Context"应该对应"风控仪表盘"页面，也不知道"Device Aggregate"应该对应"设备列表+详情"页面结构
无架构关联：它生成的线框图是独立产物，无法反向验证是否覆盖了 C4 Component 的所有接口
arsitect 的自研方案：
Agent 1: DomainMapper：读取 domain_model.json + c4_container.json，映射领域实体到页面（如 Device → 设备管理页：列表/详情/编辑）
Agent 2: LayoutPlanner：基于页面类型（列表页/详情页/仪表盘）选择布局模式，生成 SVG 坐标
Agent 3: NavigationLinker：根据 C4 接口依赖关系，建立页面跳转（如"点击设备 → 跳转设备详情 → 调用 GET /api/v1/devices/{id}"）
这样线框图天生与架构模型绑定，评审时可以直接检查"这个页面是否漏了某个接口"。
六、C4 InterFlow 的引入策略：融合
C4 InterFlow 是 arsitect 必须融合的基础设施，因为它填补了 arsitect 架构设计中最硬核的空白：
表格
arsitect 需求	C4 InterFlow 能力	融合方式
C4 模型即代码	YAML/JSON DSL 定义架构	直接采用其 DSL 作为 arsitect-aac 格式
从架构生成图	CLI draw-diagrams 生成 PlantUML/SVG/PNG	封装为 C4DiagramEngine 服务
架构查询分析	JSONPath-like 查询依赖关系	集成到 ArchitectureValidator
CI/CD 发布	GitHub Actions 自动渲染	直接复用其流水线模板
融合方式：不是 Fork 改造，而是作为依赖库/CLI 工具集成到 arsitect 的 ArchitectureGenerator 节点中。arsitect 生成 YAML 格式的架构模型，调用 C4 InterFlow CLI 渲染成图。

二、调整 1：架构模型层 —— 全面采用 C4 InterFlow DSL
2.1 为什么必须换底座？
C4 InterFlow 不是「又一个画图工具」，而是架构即代码（Architecture as Code）的基础设施。它提供了 arsitect 最缺失的硬核能力：
表格
C4 InterFlow 能力	arsitect 之前缺失	补全后的价值
DSL 定义架构	用自定义 JSON 描述 C4，无标准	采用社区标准 AaC，可扩展、可校验
从代码反向生成	架构→代码单向流动	代码生成后，反向扫描生成 C4 图，做架构漂移检测
架构查询引擎	无法做影响分析	$.components[?(@.technology=='Spring Boot')] 查询依赖
CLI 批量渲染	依赖 PlantUML 在线服务	本地 CLI 生成 SVG/PNG/Mermaid，CI 友好
多语言扩展	只支持概念层	可扩展扫描 Java/Go/Python 源码自动生成 Component 图

三、调整 2：编排层 —— 引入 Coco 式复杂度路由
3.1 为什么固定流水线是浪费？
Coco Workflow 的核心洞察：不是所有需求都值得完整仪式。一个字段修改和一个新微服务不应该走同样的流程。
3.2 复杂度路由矩阵
表格
复杂度	触发条件	arsitect 执行路径	跳过的阶段
Trivial	单文件修改、bug fix、字段调整	hotfix → 直接生成 Context Package → 调用 AI Code 平台	跳过 DomainParser、C4 建模、原型验证
Light	1-3 文件、单页面、单接口	design+import → 轻量架构草图（C4 Container 一层）→ 直接生成	跳过深度技术调研、兼容性验证
Standard	新模块、多接口、有状态机	完整流水线：Domain → C4 Context → C4 Container → TechStack → API 契约 → 原型验证	无跳过
Deep	新领域、核心系统重构、跨服务架构	完整流水线 + C4 InterFlow 架构查询 + OpenHands 沙箱验证 + 架构漂移检测	增加反向验证
3.3 路由决策器实现
Python
# arsitect/orchestrator/complexity_router.py
class ComplexityRouter:
    def route(self, requirement: Requirement) -> ExecutionPath:
        signals = self.analyze_signals(requirement)
        
        if signals.file_count <= 1 and signals.no_new_entities:
            return HotfixPath()  # Trivial
            
        elif signals.file_count <= 3 and signals.single_bounded_context:
            return LightPath()   # Light，C4 Container 一层即可
            
        elif signals.has_new_domain or signals.cross_service:
            return DeepPath()    # Deep，完整+反向验证
            
        else:
            return StandardPath() # Standard
四、调整 3：执行层 —— 引入 OpenHands 作为沙箱执行器
4.1 OpenHands 的不可替代性
OpenHands（原 OpenDevin）的 CodeAct 架构 将动作空间统一为代码执行，在 Docker 沙箱中自主完成：
文件读写（Edit/Write）
命令执行（Bash）
浏览器操作（Browser）
代码搜索（Grep）
SWE-bench 验证性能 77%，这意味着它在真实代码库上解决 issue 的能力已经成熟。
4.2 与现有执行器的对比
表格
执行器	自动化等级	安全隔离	适用场景	arsitect 角色
Claude Code	L5	❌（本地文件系统）	开发者本地快速迭代	首选日常执行器
Aider	L5	❌（本地 Git 仓库）	批量修改、CI 集成	批量重构执行器
OpenHands	L5	✅（Docker 沙箱）	无人值守、高风险操作、架构验证	安全沙箱执行器

validation_rules:
  - name: "所有微服务必须有数据库"
    query: "$.structures[?(@.type=='container' && @.tags contains 'microservice')]"
    assert: "each has interface to type='database'"
    
  - name: "核心领域必须有 API 契约"
    query: "$.structures[?(@.tags contains 'core-domain')]"
    assert: "each has interfaces with type='http'"
    
  - name: "无循环依赖"
    query: "cycles in flows"
    assert: "count == 0"

jobs:

    runs-on: ubuntu-latest

      - uses: actions/checkout@v4

2.1 C4 InterFlow —— 依赖 CLI + 融合 DSL（最深融合）
为什么不 Fork 改造？
C4 InterFlow 是 .NET 生态的 CLI 工具，arsitect 大概率是 Python/Node 技术栈。Fork 后维护跨语言代码成本极高。MIT 协议允许直接调用其发布产物。
具体集成三层：
表格
层级	实现方式	代码示例
DSL 生成层	arsitect 输出符合 C4 InterFlow 的 YAML	arsitect.aac.yml（见下文）
CLI 调用层	subprocess / Docker 调用 c4-interflow 命令	subprocess.run(["c4-interflow", "draw-diagrams", "--aac-file", ...])
反向扫描层	代码生成后，调用 reverse-engineer 生成实际架构	subprocess.run(["c4-interflow", "reverse-engineer", "--source", "src/"])

2.2 Coco Workflow —— 纯借鉴架构（零代码引入）
为什么零代码？
Coco Workflow 是 Claude Code 专属插件（基于 claude-code 的 slash command 体系）。直接引入会让 arsitect 绑定到 Claude Code 生态，丧失平台中立性。
借鉴什么？
只提取其复杂度信号检测和路由决策表：
Python
# arsitect/orchestrator/complexity_router.py
# 自研实现，借鉴 Coco 的信号-路径映射思想

class ComplexityRouter:
    """
    借鉴 Coco Workflow 的复杂度路由思想：
    - Trivial: 单文件、无新实体 → hotfix
    - Light: 1-3文件、单边界上下文 → 轻量架构
    - Standard: 多模块、有依赖 → 完整流水线
    - Deep: 新领域、跨服务 → 完整+反向验证
    """
    
    def analyze_signals(self, requirement: Requirement) -> ComplexityLevel:
        signals = {
            "file_count": requirement.estimated_files,
            "new_entities": requirement.new_domain_entities,
            "bounded_contexts": len(requirement.affected_contexts),
            "cross_service": requirement.has_cross_service_dependency,
            "has_state_machine": requirement.has_state_machine,
            "has_new_tech": requirement.introduces_new_technology
        }
        
        if signals["file_count"] <= 1 and not signals["new_entities"]:
            return ComplexityLevel.TRIVIAL  # 对应 Coco 的 hotfix
            
        elif signals["file_count"] <= 3 and signals["bounded_contexts"] == 1:
            return ComplexityLevel.LIGHT    # 对应 Coco 的 design+import
            
        elif signals["cross_service"] or signals["has_new_tech"]:
            return ComplexityLevel.DEEP     # 对应 Coco 的 interview+design+tasks
            
        else:
            return ComplexityLevel.STANDARD   # 对应 Coco 的 standard pipeline
借鉴点：信号权重、阈值设定、路径选择逻辑。代码完全自研。

# arsitect/executors/openhands_executor.py
import requests

class OpenHandsExecutor:
    def __init__(self, api_url: str):
        self.api_url = api_url
        
    def execute(self, context_package: ContextPackage) -> ExecutionResult:
        """
        发送 Context Package 给 OpenHands，在 Docker 沙箱中执行
        """
        payload = {
            "task": self.assemble_prompt(context_package),
            "workspace_mount_path": context_package.workspace_path,
            "config": {
                "agent": "CodeActAgent",
                "max_iterations": 100,
                "sandbox": {"type": "docker"}
            }
        }
        
        # 启动会话
        session = requests.post(f"{self.api_url}/sessions", json=payload).json()
        session_id = session["session_id"]
        
        # 轮询结果
        result = self.poll_until_complete(session_id)
        
        # 提取生成的文件列表 + 执行日志
        return ExecutionResult(
            files=result["files"],
            logs=result["history"],
            success=result["status"] == "completed"
        )
关键设计：OpenHands 只操作挂载目录（workspace），arsitect 把 Context Package 和项目骨架写入挂载目录，OpenHands 在沙箱中生成代码，arsitect 读取结果做架构漂移检测。
2.4 GTPlanner —— 纯借鉴架构（零代码引入）
为什么不引入 PocketFlow 源码？
GTPlanner 的 PocketFlow 是通用异步工作流引擎，节点语义是 short_planning、tool_recommend（通用规划）。arsitect 需要的是 DomainParser、C4ModelManager、ContractDesigner（软件架构语义）。强行适配会造成功能扭曲。
借鉴什么？
只提取其节点生命周期协议和状态驱动流转：
Python
# arsitect/core/architect_node.py
# 借鉴 GTPlanner / PocketFlow 的 prep → exec → post 三阶段

from abc import ABC, abstractmethod
from typing import Dict, Any, Literal

class ArchitectNode(ABC):
    """
    借鉴 GTPlanner PocketFlow 的 Node 抽象：
    - prep: 从 ProjectState 读取输入，做预处理
    - exec: 执行核心架构计算（LLM 调用 / 检索 / 渲染）
    - post: 更新 ProjectState，返回 action 决定下一节点
    """
    
    @abstractmethod
    def prep(self, shared: ProjectState) -> PrepResult:
        """读取共享状态，返回预处理结果"""
        pass
    
    @abstractmethod
    def exec(self, prep_res: PrepResult) -> ExecResult:
        """执行核心计算，返回执行结果"""
        pass
    
    @abstractmethod
    def post(self, shared: ProjectState, prep_res: PrepResult, exec_res: ExecResult) -> Literal["proceed", "feedback", "block", "drift"]:
        """
        更新共享状态，返回 action：
        - proceed: 进入下一节点
        - feedback: 回退到当前节点重新执行
        - block: 人工闸门拦截
        - drift: 架构漂移，触发反向验证
        """
        pass
    
    def run(self, shared: ProjectState) -> Literal["proceed", "feedback", "block", "drift"]:
        """标准运行模板，借鉴 PocketFlow 的 Node.run()"""
        try:
            prep_res = self.prep(shared)
            exec_res = self.exec(prep_res)
            action = self.post(shared, prep_res, exec_res)
            return action
        except Exception as e:
            return self.exec_fallback(shared, e)
    
    def exec_fallback(self, shared: ProjectState, error: Exception) -> Literal["block"]:
        """降级处理：异常时直接拦截"""
        shared.errors.append(str(error))
        return "block"
借鉴点：三阶段分离、异常降级、action 驱动流转。实现完全自研，不引入 PocketFlow 依赖。
2.5 OpenUI —— 依赖服务（Docker + HTTP）
为什么不借鉴核心代码？
OpenUI 的核心是"自然语言 → 多框架前端代码 → 实时预览"，涉及复杂的 LLM Prompt Engineering、组件库映射、代码沙箱渲染。自研成本极高，且 arsitect 不需要掌握前端生成细节。
集成方式：

# arsitect/services/openui_adapter.py
import requests

class OpenUIAdapter:
    def __init__(self, endpoint: str = "http://openui:7878"):
        self.endpoint = endpoint
        
    def generate_prototype(self, page_spec: PageSpec) -> PrototypeResult:
        """
        将 arsitect 的页面规范转换为 OpenUI 提示词
        """
        prompt = f"""
        Generate a {page_spec.framework} page for: {page_spec.name}
        
        Components: {', '.join(page_spec.components)}
        Data flow: {page_spec.data_flow}
        Style: {page_spec.theme}
        
        Must use these API endpoints: {page_spec.api_endpoints}
        """
        
        response = requests.post(
            f"{self.endpoint}/api/generate",
            json={"prompt": prompt, "format": "html"}
        )
        
        return PrototypeResult(
            html=response.json()["html"],
            react_code=response.json().get("react"),
            preview_url=f"{self.endpoint}/preview/{response.json()['id']}"
        )

2.6 AI Wireframe Generator —— 借鉴核心模式 + 自研（算法级）
为什么部分移植？
AI Wireframe Generator 的 LangGraph StateGraph 构建模式（多 Agent 流水线）是通用算法，MIT 允许借鉴。但其节点实现（通用布局规划）不满足 arsitect 的领域感知需求。
移植范围：
移植：StateGraph 的构建逻辑、节点连接方式、状态传递机制
自研：节点内部实现（DomainMapper、LayoutPlanner、NavigationLinker）
Python
# arsitect/prototype/wireframe_engine.py
# 借鉴 AI Wireframe Generator 的 LangGraph 多 Agent 架构，自研领域节点

from langgraph.graph import StateGraph, END

class WireframeEngine:
    """
    借鉴 AI Wireframe Generator 的 LangGraph 多 Agent 流水线：
    - 需求解析 Agent → 布局规划 Agent → SVG 渲染 Agent
    但节点实现完全自研，注入领域知识
    """
    
    def build_graph(self):
        builder = StateGraph(WireframeState)
        
        # Agent 1: 领域映射（自研）
        builder.add_node("domain_mapper", self.domain_mapper)
        
        # Agent 2: 布局规划（自研）
        builder.add_node("layout_planner", self.layout_planner)
        
        # Agent 3: 导航连接（自研）
        builder.add_node("navigation_linker", self.navigation_linker)
        
        # Agent 4: SVG 渲染（自研）
        builder.add_node("svg_renderer", self.svg_renderer)
        
        # 借鉴其线性流水线连接方式
        builder.add_edge("domain_mapper", "layout_planner")
        builder.add_edge("layout_planner", "navigation_linker")
        builder.add_edge("navigation_linker", "svg_renderer")
        builder.add_edge("svg_renderer", END)
        
        return builder.compile()
    
    def domain_mapper(self, state: WireframeState):
        """
        自研：读取 C4 InterFlow DSL，将 Container/Component 映射到页面
        例如：RiskControlService → "风控仪表盘" 页面
        """
        c4_model = load_c4_model(state.project_id)
        state.pages = self.map_containers_to_pages(c4_model.structures)
        return state
关键：只借鉴其编排模式（LangGraph 状态机），不借鉴其业务逻辑（通用布局算法）。
