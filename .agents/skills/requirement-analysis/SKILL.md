# Requirement Analysis

## 适用场景
- 收到模糊的产品需求，需要拆解为可执行的任务
- 敏捷迭代前的用户故事梳理
- 客户需求澄清会议前的准备
- 评估需求变更的影响范围

## 核心职责
1. 识别需求中的模糊点和隐藏假设
2. 将大需求拆分为独立、可测试的用户故事
3. 为每个用户故事定义明确的验收标准（AC）
4. 识别技术风险和依赖项
5. 估算工作量（T 恤尺码或故事点）

## 执行步骤

### 阶段 1：输入分析（Input Analysis）

**Gate In — 输入合法性检查**：
1. **Schema 校验**：确认输入包含以下至少一项：
   - 用户原始需求描述（直接输入或引用文档）
   - 上游 skill 传递的结构化需求（如 `brainstorming/requirement-draft.md`）
   - 现有需求文档路径（`@路径`）
2. **来源识别**：区分并记录来源类型：
   - `user_direct`：用户直接输入的模糊需求
   - `upstream_skill`：`brainstorming`、`prd-generation` 等上游产出
   - `external_doc`：用户提供的本地文档、会议纪要、邮件等
3. **来源溯源表**：在分析报告中记录来源清单，确保后续覆盖验证有迹可循
4. **异常拦截**：输入为空或仅包含无意义内容时，终止执行并提示用户提供具体需求

### 阶段 2：条目整理与执行

## 分析框架

### 5W2H 检查
- **What**：需求的核心功能是什么？
- **Why**：解决什么痛点？价值主张是什么？
- **Who**：目标用户是谁？角色（Persona）是什么？
- **When**：时间约束？是否有发布窗口？
- **Where**：使用场景？平台（Web/App/小程序）？
- **How**：用户如何操作？核心流程是什么？
- **How much**：数据量？并发量？预算约束？

### 用户故事模板
```
作为 <角色>，
我希望 <目标>，
以便 <价值>。
```

### 验收标准（Given-When-Then）
```
Given <前置条件>
When <用户操作>
Then <预期结果>
```

### 阶段 4：覆盖验证（Coverage Validation）

**Gate Out — 输出完整性检查**：
在输出最终需求分析报告前，必须执行以下检查：

1. **条目回溯**：对照阶段 1 记录的「来源溯源表」，逐条验证用户原始需求是否已映射到输出：
   - 用户直接提出的每个功能点 / 痛点 / 约束，必须在用户故事、验收标准或澄清问题中有所回应
   - 上游 skill 传递的业务规则、数据假设、范围声明，必须在当前报告中继承或显式说明变更理由
   - 外部文档（`@路径`）中的核心条款，必须在报告中标注引用位置
2. **遗漏清单**：若发现来源条目未覆盖，生成待补充项列表，标记为 🔴 BLOCKER 或 🟡 WARNING
3. **幻觉检测**：报告中的事实性声明（如"用户规模为 X"、"行业平均 Y"）必须标注来源；无法溯源的标记为 `[待验证]`
4. **一致性校验**：检查输出内部是否存在矛盾（如用户故事与业务规则冲突、验收标准与范围声明矛盾）

> **禁止交付条件**：存在未解决的 🔴 BLOCKER 级遗漏时，禁止输出最终报告，必须返回补充分析。

## 输出格式

### 需求分析报告
```markdown
## 需求概述
- 背景：...
- 目标：...
- 干系人：...

## 澄清问题（Clarifying Questions）
1. ...？（影响范围/优先级）
2. ...？（技术约束）

## 用户故事拆分

### US-1: ...
- **故事**：作为...，我希望...，以便...
- **验收标准**：
  - [ ] AC1: Given... When... Then...
  - [ ] AC2: ...
- **工作量**：M（2-3 天）
- **依赖**：US-X
- **风险**：...

## 技术风险
1. ...

## 建议优先级
P0: US-1, US-2
P1: US-3
```

## 示例

### 示例 1：电商需求分析

**原始需求**：
> "用户想要一个购物车，可以加东西，最后能付款。"

**分析输出**：

#### 澄清问题
1. 购物车是否支持跨设备同步（登录态）？
2. 支持哪些支付方式（微信/支付宝/信用卡）？
3. 商品库存是实时扣减还是下单时扣减？
4. 是否需要支持优惠券和满减活动？

#### 用户故事拆分

**US-1: 添加商品到购物车**
- 故事：作为顾客，我希望将商品加入购物车，以便稍后统一结算。
- AC：
  - Given 商品有库存，When 点击「加入购物车」，Then 购物车数量 +1
  - Given 商品无库存，When 点击「加入购物车」，Then 提示「暂时缺货」

**US-2: 购物车结算**
- 故事：作为顾客，我希望从购物车发起结算，以便完成购买。
- AC：
  - Given 购物车中有商品，When 点击「去结算」，Then 进入订单确认页
  - Given 购物车为空，When 点击「去结算」，Then 提示「购物车是空的」

## 注意事项
- 不要假设用户已知的信息，主动列出需要确认的问题
- 拆分粒度以「一个迭代内可完成」为宜
- 区分「必须做（Must have）」和「锦上添花（Nice to have）」
