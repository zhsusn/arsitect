# 单模块深度模式指南

> 本指南是 `detailed-requirements` 的备选执行模式。当项目规模较小、模块数量 ≤3，或某个核心模块需要穷尽式人工协作时使用。

---

## 何时使用单模块深度模式

| 场景 | 推荐模式 |
|------|----------|
| 模块数量 ≥4，需要标准化批量输出 | **默认模式**（4 文件：spec/prototype/io-table/logic） |
| 模块数量 ≤3，或仅 1 个核心模块需要深挖 | **单模块深度模式**（本指南） |
| 模块业务逻辑极复杂，需要逐分支人工确认 | **单模块深度模式** |
| 已有 PRD-000 基线，仅需为单个模块补详细需求 | **单模块深度模式** |

---

## 与默认模式的区别

| 维度 | 默认模式（批量） | 单模块深度模式 |
|------|------------------|----------------|
| 输出结构 | 4 文件标准化目录 | 单文档 PRD-00X（11 章节） |
| 交互方式 | 系统化串行生成 | "无情访谈"，逐分支人工确认 |
| 校验重点 | Cross-Module 一致性 | 实体/Scope/NFR 分层合规 |
| 适用规模 | 中大型项目 | 小型项目或单模块深挖 |

---

## 执行步骤

### Step 1：继承校验（Baseline Inheritance）
1. 读取当前模块在 PRD-000 中的定位、Scope 边界和 NFR 约束
2. 复述约束并向用户确认
3. 声明：本详细 PRD 不得引入 PRD-000 未定义的新实体，不得修改已冻结的 Scope 和 NFR

### Step 2：功能树构建（Feature Tree Construction）
将模块拆解为功能分支树，按优先级排序（P0 优先）。每个叶子节点对应一个用户故事或独立页面。

### Step 3：深度访谈（Relentless Interview）
对每个功能分支（从 P0 开始），逐个确认：
1. **用户故事**：格式 `作为 <角色>，我希望 <目标>，以便 <价值>`
2. **业务流程**：绘制文本流程图（Mermaid 或步骤列表），标注决策节点和异常分支
3. **业务逻辑四要素**：输入、处理逻辑、输出、副作用
4. **页面原型与交互**：布局、元素、状态变化、字段清单
5. **接口契约**：模块间接口方法名、入参、出参、错误码；外部 API 调用及降级策略
6. **权限与兼容性**：RBAC 矩阵、旧数据兼容
7. **分支确认**：完成一个分支后展示完整内容，用户确认无误再进入下一分支

### Step 4：分层合规检查
1. **实体合规**：所有数据实体必须在 PRD-000 核心实体列表中能找到
2. **Scope 合规**：详细功能全部落在 PRD-000 In-Scope 内
3. **NFR 合规**：详细指标只细化、不推翻全局 NFR
4. **版本引用**：文档开头正确引用 PRD-000 版本号

### Step 5：质量门禁
1. **模糊语言扫描**：替换"优化""完善""提升""友好"等模糊词为量化描述
2. **验收标准可测性**：每个 AC 必须包含 Given-When-Then，预期结果可自动化测试或人工核验
3. **任务分解可行性**：用户故事粒度在一个迭代内可完成

## 输出格式

按 `feature-template.md` 输出单文档 PRD-00X：

```markdown
# PRD-00X {模块名称} 详细需求
> 隶属概要：PRD-000 v{X.X}

## 1. 功能概述
## 2. 用户故事与验收标准
## 3. 业务流程图
## 4. 业务逻辑（输入 / 处理 / 输出 / 副作用 / 异常）
## 5. 页面原型与交互
## 6. 数据需求
## 7. 接口契约
## 8. 异常与边界场景
## 9. 权限矩阵（RBAC）
## 10. 性能与兼容性
## 11. 验收 Checklist
```

详细模板见同目录 `feature-template.md`。
