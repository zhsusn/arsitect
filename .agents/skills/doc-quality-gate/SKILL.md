# Doc Quality Gate

AI 生成文档的多文档一致性检查与自动修复 Skill。解决 AI 生成多文档时的系统性质量漂移问题：单文档内部错误、文档间交叉不一致、规范合规性缺失。

## 适用场景

- brainstorming → prd-generation → architecture-decision 等阶段之间的文档健康检查
- 单文档内部：计算错误、术语混用、状态标记混乱、章节缺失
- 文档间交叉：同一数据在不同文档中口径不一致、引用悬空、编号断层
- 规范合规：违反单一事实来源原则、假设登记册遗漏、边界范围标记错误

## 检查维度（三层九项）

### Layer 1: 完整性（Completeness）

| 检查项 | 检查内容 | 严重级别 | 自动修复 |
|--------|----------|----------|----------|
| C1-结构完整性 | 必需章节是否齐全 | 🔴 阻断 | ❌ 需人工补充 |
| C2-数据完整性 | 表格计算值是否与原始数据一致 | 🔴 阻断 | ✅ 自动重算修正 |
| C3-引用完整性 | 交叉引用是否指向存在的章节/文档/编号 | 🟡 警告 | ❌ 需人工确认 |
| C4-列表完整性 | 枚举项是否完整 | 🟡 警告 | ❌ 需人工补充 |

### Layer 2: 不一致性（Consistency）

| 检查项 | 检查内容 | 严重级别 | 自动修复 |
|--------|----------|----------|----------|
| I1-数值一致性 | 同一数值在不同文档/表格中的声明是否一致 | 🔴 阻断 | ✅ 自动对齐 |
| I2-术语一致性 | 同一概念是否有多种表述 | 🟡 警告 | ✅ 自动统一 |
| I3-状态一致性 | 状态标记是否符合阶段约定 | 🔴 阻断 | ✅ 自动替换 |
| I4-规则一致性 | 映射规则与执行结果是否一致 | 🔴 阻断 | ✅ 自动修正 |

### Layer 3: 规范性（Compliance）

| 检查项 | 检查内容 | 严重级别 | 自动修复 |
|--------|----------|----------|----------|
| P1-单一事实来源 | 假设登记册是否为唯一事实来源 | 🟡 警告 | ✅ 自动编号并入登记册 |
| P2-编号连续性 | 风险 ID、假设 ID 是否连续 | 🟡 警告 | ✅ 自动补全关联列 |
| P3-格式规范 | Markdown 表格对齐、标题层级、代码块语法 | 🟢 提示 | ✅ 自动格式化 |
| P4-版本一致性 | 文档头中的版本、生成时间是否一致 | 🟢 提示 | ✅ 自动同步 |
| **P5-YAML Front Matter 合规（V3.1 新增）** | 字段完整性、`doc_type` 合法性、`c4_binding` 强制存在性、YAML 语法正确性 | 🔴 阻断 | ❌ 需人工修正 |
| **P6-锚点ID 规范（V3.1 新增）** | `##`/`###` 标题是否含 `{#sec-xxx}`、格式合法性、同文档唯一性 | 🟡 警告 | ✅ 自动补全 |
| **P7-C4 绑定一致性（V3.2 修订）** | `c4_binding` 中定义的架构元素与文档末尾《C4 标签映射表》是否双向对应 | 🟡 警告 | ❌ 需人工确认 |
| **P8-依赖声明完整性（V3.1 新增）** | `dependencies` 数组格式、`fragment_id` + `version` 是否填充、上游文档是否存在 | 🟡 警告 | ❌ 需人工补充 |

## 执行流程（五阶段流水线）

1. **Phase 1: 文档解析与索引** — 提取所有数值、术语、状态标记、引用链接、编号、表格结构；**V3.1 新增**：解析 YAML Front Matter 元数据、提取锚点ID 列表、`c4_binding` 架构元素、`dependencies` 依赖图谱
2. **Phase 2: 单文档检查** — 结构完整性、计算校验、格式规范；**V3.1 新增**：YAML Front Matter 字段合法性、`doc_type`↔`c4_binding.level` 一致性、锚点ID 规范；**V3.2 修订**：C4 标签映射表与 `c4_binding` 双向一致性（不再检查正文内联 `@C4-` 标签）
3. **Phase 3: 跨文档检查** — 构建单一事实来源索引，执行数值/术语/状态/规则一致性检查；**V3.1 新增**：`dependencies` 中引用的上游文档是否存在、版本号是否与上游当前 `version` 一致
4. **Phase 4: 自动修复** — 对标记为"✅ 自动修复"的问题执行修复
5. **Phase 5: 质量报告** — 生成质量检查报告 + 待人工项清单 + 修复日志 YAML

## 修复规则引擎

### 数值对齐规则
- 以详细计算表为基准，修正综合评估表中的总分
- 如因计算修正导致选型改变，同步修正并标注"因计算修正导致选型调整"

### 状态标记规范化
- COMPLETED → INCLUDED 或 EXCLUDED（需求阶段禁用 COMPLETED）
- NOT_STARTED → EXCLUDED 或 PLANNED

### 术语统一
- 以关键决策表或首次定义位置为基准
- 统计各拼写出现频次，全局替换其他拼写（保留代码块、引用链接不变）

### 假设登记册同步
- 为每条新假设分配 A-008~A-0xx 编号
- 将假设内容、置信度、推翻条件、Plan B 提取并追加到登记册
- 在业务假设章节添加引用

## 输出格式

### 质量检查报告（Markdown）

包含：
- 摘要统计表（阻断/警告/提示/已自动修复/需人工处理）
- 阻断级问题清单（必须修复）
- 警告级问题清单（建议修复）
- 提示级问题清单（可选优化）
- 待人工确认项

### 修复日志（YAML）

```yaml
doc_quality_gate:
  version: "1.0.0"
  run_id: "{project_name}-{timestamp}"
  documents_checked: [...]
  auto_fixes: [...]
  manual_fixes_required: [...]
  pending_items: [...]
```

## 扩展规则

用户可在项目根目录创建 `.doc-quality-gate.yml` 覆盖默认规则：

```yaml
project: sdlc-visualizer
terminology:
  baseline_doc: "requirement-draft.md"
  terms:
    - canonical: "arsitect"
      aliases: ["arsitecT", "ArsitecT"]
status_mapping:
  requirement_phase:
    allowed: ["INCLUDED", "PLANNED", "EXCLUDED"]
    forbidden: ["COMPLETED", "DONE"]
calculation_rules:
  - name: "加权评分"
    doc_pattern: "ai-architecture-decision.md"
    formula: "sum(维度评分 * 权重)"
    precision: 2
required_sections:
  "requirement-draft.md":
    - "客户叙事"
    - "核心问题"
    - "边界范围"
```

## 与 self-check 的关系（分工边界）

`doc-quality-gate` 与 `self-check` 是互补关系，**不可相互替代**：

| 维度 | `doc-quality-gate` | `self-check` |
|------|-------------------|-------------|
| 职责 | **文档修复**：多文档一致性检查与自动修复 | **阶段门控**：全生命周期产出物审查 |
| 行为 | **可自动修复**数值/术语/状态/格式问题 | **只读**，不修改任何文件 |
| 范围 | 仅文档（Markdown / YAML） | 文档 + 代码 + 测试 + UAT + 发布 |
| 输出 | Markdown 报告 + YAML 修复日志 | Markdown 自查报告 |

**协作流程：**
1. 阶段完成后，**先执行 `self-check`** 做广度门控
2. 若 `self-check` 发现文档类 BLOCKER，**调用 `doc-quality-gate`** 执行深度检查与自动修复
3. `doc-quality-gate` 完成自动修复后，**必须重新执行 `self-check`** 确认该阶段通过，方可进入下一阶段

**定位说明：**
- `doc-quality-gate` **不是阶段门控**，不检查代码接口一致性、测试覆盖率、UAT 缺陷记录质量等非文档内容
- `doc-quality-gate` 的 YAML 修复日志可被下游 Skill 或脚本解析，但**不能替代 `self-check` 的 BLOCKER/WARNING/INFO 分级门控判断**

## Gotchas

- **自动修复不覆盖内容创作**：缺失的章节、未定义的术语、待确认的业务规则只能标记位置，不能凭空生成内容
- **数值修复以计算为准**：如原始计算公式本身错误，Skill 会按错误公式"一致地"修复，需人工复核公式正确性
- **术语统一可能误伤**：代码块、引用链接、外部系统名称中的相似拼写可能被误替换，修复后需人工抽查
- **多文档一致性依赖完整输入**：如用户未提供全部被引用文档，跨文档检查可能产生误报（如引用悬空）
- **阻断级问题必须清零**：存在 🔴 阻断级问题时，禁止进入下一阶段
- **修复后必须重跑**：自动修复后需重新执行 doc-quality-gate 确认问题已解决
- **YAML 日志供下游消费**：修复日志采用结构化 YAML，下游 Skill 可直接解析
- **自定义规则优先**：`.doc-quality-gate.yml` 存在时，以用户自定义规则覆盖内置默认规则
