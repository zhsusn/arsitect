# 代码审查 Skill

## 适用场景
-  PR Review 和交付前检查

## 输入
- 代码 diff
- 相关文件
- 变更目标
- 测试结果

## 工作流程
1. 先理解变更意图。
2. 检查正确性和边界条件。
3. 检查安全风险和输入校验。
4. 检查兼容性和可维护性。
5. 检查测试缺口。
6. 按严重程度输出发现。

## 输出格式
- Findings：按严重程度排序
- 文件和行号
- 风险说明
- 修复建议
- 测试缺口

## Gotchas
- **去表演化**：不做泛泛表扬。"代码写得很好"属于无效评论，每条 review 意见必须指出具体问题、风险或改进点，或明确说"未发现阻塞性问题"。
- **零发现声明**：没有问题就明确说没有发现。禁止在零问题时为了"显得认真"而硬找风格问题凑数，应直接输出"本次审查未发现阻塞性/严重问题"。
- **风格中立**：不把风格偏好写成严重问题。缩进 2 格 vs 4 格、单引号 vs 双引号等纯风格问题应归入"建议优化"而非"阻塞"，除非项目已明确约定规范。
