# Monitoring Analysis（线上监控分析）

## 适用场景

- `finish` 归档完成后，进入周期性运维阶段
- 用户要求"生成本周健康报告"、"监控分析"、"线上问题复盘"
- 需要基于监控规则基线评估系统运行状态
- 需要产出 `feedback-loop.md` 反哺下一变更的 `brainstorming`
- 告警触发后，需要分析问题根因和影响范围

## 角色

你是项目运维健康分析师。周期性读取监控规则基线与运行时数据，评估系统健康度、识别潜在风险、提炼产品改进建议。

## 前置依赖

| 依赖项 | 路径 | 说明 |
|--------|------|------|
| 监控规则基线 | `ops/monitoring-rules.yaml` | 由 monitoring-setup 生成，定义告警阈值与采集目标 |
| 归档完成确认单 | `openspec/changes/{变更名}/archive-confirmation.md` | 确认变更已上线并可纳入监控 |
| 运行时数据 | 外部系统（Sentry/Prometheus/Grafana/云监控） | 错误率、延迟、QPS、资源使用率 |
| 埋点数据 | 外部系统（如神策/GrowingIO/自建） | 用户行为、业务转化漏斗 |

## 执行流程

### Phase 1: 读取基线与数据

1. 读取 `ops/monitoring-rules.yaml`：
   - 提取 `alerting_rules` 中的阈值定义（P99 延迟、错误率、CPU、内存等）
   - 提取 `business_metrics` 中的业务埋点定义
   - 提取 `scrape_configs` 中的服务列表，确认监控覆盖范围
2. 读取运行时数据（由用户提供或 AI 辅助查询）：
   - 基础设施层：CPU、内存、磁盘、网络
   - 应用层：QPS、P50/P99 延迟、错误率、吞吐量
   - 业务层：核心链路转化率、活跃用户数、订单量
   - 告警历史：本周触发告警的次数、级别、恢复时间
3. 读取最近的 `archive-confirmation.md` 和 `release-notes.md`，确认当前线上版本信息

## 与上下游衔接

| 衔接点 | 动作 |
|--------|------|
| 上游: finish | 归档完成后触发首次监控分析；后续周期性执行 |
| 上游: monitoring-setup | 消费 `ops/monitoring-rules.yaml` 作为分析基准和阈值参照 |
| 下游: brainstorming | `feedback-loop.md` 作为下一变更的需求输入 |
| 横向: self-check | 检查监控数据完整性和阈值合理性 |

## 执行频率建议

| 场景 | 频率 | 触发方式 |
|------|------|----------|
| 常规巡检 | 每周一次 | 人工触发 `/skill:monitoring-analysis 生成本周健康报告` |
| 迭代收尾 | 每迭代一次 | finish 完成后自动触发 |
| 告警响应 | 事件驱动 | 告警触发后人工触发，执行根因分析 |
| 版本发布后 | 发布后 24h / 72h | 人工触发，验证新版本稳定性 |

## Gotchas

- **数据获取限制**：AI 可能无法直接访问生产环境的 Prometheus/Grafana/Sentry。执行此 skill 时，用户需提供运行时数据截图、导出文件或关键指标摘要。AI 的职责是**分析数据**而非**采集数据**。
- **阈值漂移风险**：monitoring-rules.yaml 中的阈值是设计阶段确定的基线，实际运行中可能出现"阈值合理但业务已变化"的情况。健康报告中应标注"阈值最后更新时间"，建议每季度回顾一次阈值。
- **不要过度解读单次波动**：环比变化在 ±5% 范围内通常属于正常波动，不要轻易标记为风险。连续两周同向趋势才值得预警。
- **feedback-loop.md 的时效性**：feedback-loop.md 应覆盖最近 2-4 周的监控周期，过期的反馈应归档到 `openspec/feedback/archive/`，避免 brainstorming 阶段读取过时信息。
- **与 monitoring-setup 的互补性**：monitoring-setup 是"规则生成"（一次性），monitoring-analysis 是"规则消费与调优建议"（周期性）。若发现阈值长期不适用，应在 feedback-loop.md 中提出"建议调整 monitoring-rules.yaml 中 X 项阈值"，但不直接修改 YAML。
- **业务层数据优先级**：相比基础设施层的 CPU/内存，业务层的转化率、支付成功率、用户留存更能反映产品健康度。健康报告的结论应以业务层指标为首要权重。
- **告警疲劳识别**：若某条告警一周内触发超过 3 次且均为误报，应在 feedback-loop.md 中标记为"告警疲劳"，建议上调阈值或优化告警条件。
- **闭环验证**：feedback-loop.md 中的改进建议被采纳并进入下一变更后，应在该变更上线后的 monitoring-analysis 中验证问题是否解决，形成 PDCA 闭环。
