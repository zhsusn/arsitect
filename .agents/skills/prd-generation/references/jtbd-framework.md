# JTBD + 用户故事框架指南

> 参考来源：johnnychauvet/prd-skill 的 JTBD 框架与用户故事体系。用于确保需求表达从"为什么"出发，而非从"做什么"出发。

---

## 1. Jobs to Be Done（JTBD）格式

### 标准格式

```
When [situation], I want to [motivation], so I can [expected outcome].
```

### 填写规范

| 部分 | 说明 | 示例 |
|------|------|------|
| **When [situation]** | 用户处于什么具体场景？包含时间、地点、上下文触发条件 | "When 我需要在 24 小时内交付客户演示视频" |
| **I want to [motivation]** | 用户的真实动机是什么？避免功能描述，聚焦目标 | "I want to 一键生成带配音的分镜草稿" |
| **so I can [expected outcome]** | 用户最终要达成什么结果？量化或具体化 | "so I can 节省 80% 的脚本到成片时间" |

### 反例与正例

| 反例（错误） | 正例（正确） |
|-------------|-------------|
| "用户需要一个视频编辑器" | "When 制片人收到紧急商单时, I want to 在 48 小时内完成从剧本到成片的制作, so I can 不丢单并降低外包成本。" |
| "系统要支持导出 MP4" | "When 内容团队完成剪辑后, I want to 直接导出平台适配的竖版视频, so I can 跳过格式转换步骤。" |
| "做一个 AI 写作功能" | "When 编剧卡壳时, I want to 输入 3 个关键词就获得完整的场景对白, so I can 在 10 分钟内突破创作瓶颈。" |

### 优先级规则

按 **频率 × 重要性** 排序：
1. **高频且重要**：每日发生、直接影响核心指标 → 优先级 1
2. **中频但重要**：每周发生、影响关键体验 → 优先级 2
3. **低频但重要**：每月发生、影响决策/合规 → 优先级 3
4. **低频且次要**：偶尔发生、锦上添花 → 不纳入或放入 Phase 2+

## 2. 用户故事（User Stories）

### 标准格式（扩展版）

基础格式：
```
As a [role], I want to [action], so that [benefit].
```

**概要需求阶段强制扩展**：在基础格式之上，必须补充 **应用场景（Context）** 和 **完整操作路径（Action Sequence）**，形成五段式结构：

```markdown
**应用场景（Given/When）**：
用户在什么上下文进入该场景？当前页面、已有数据、系统状态、触发事件是什么？

**用户目标（So That / Outcome）**：
用户要达成的业务结果（Outcome）是什么？注意区分 Output（功能输出）与 Outcome（业务价值）。

**完整操作路径（Action Sequence）**：
1. 用户执行动作 A
2. 系统响应 X
3. 用户执行动作 B
4. 系统响应 Y
5. ...（直至目标达成）

**即时收益（Benefit）**：
完成该路径后用户立即获得什么价值？

**关联 JTBD**：J{编号}
```

### ID 编码规则（修正）

- **概要需求阶段**（PRD-000）：使用 `US-XXX` 顺序编号（此时模块尚未拆分，无需模块前缀）
  - 示例：`US-001`、`US-002`
- **详细需求阶段**（PRD-001~PRD-00N）：使用 `US-{模块编号}-{序号}`
  - 示例：`US-001-001` 表示 feature-01 模块的第 1 条用户故事
- **跨模块通用故事**：使用 `US-COM-{序号}`

> 禁止在概要需求阶段使用 `US-{模块编号}-{序号}` 格式，避免与详细阶段混淆。

### 用户故事与 JTBD 的映射

每个用户故事必须关联至少一个 JTBD，且必须描述**完整操作路径**而非单一动作：

| ID | Role | 应用场景（Given/When） | 完整操作路径 | 用户目标（So That） | JTBD Ref | 优先级 |
|----|------|----------------------|-------------|-------------------|----------|--------|
| US-001-001 | 制片人 | Given 收到客户紧急商单，When 需要在 48 小时内交付... | 1. 上传剧本文件 → 2. 系统自动解析角色和场景 → 3. 用户确认/修正解析结果 → 4. 系统自动生成分镜草稿 | 能在极短周期内获得可调整的分镜初稿，而非从零开始绘制 | J1 | P0 |
| US-001-002 | 制片人 | Given 已生成分镜草稿，When 需要调整叙事节奏... | 1. 查看分镜时间轴 → 2. 拖拽调整镜头顺序 → 3. 修改单镜头时长 → 4. 实时预览整体节奏 → 5. 保存版本 | 能直观控制叙事节奏并即时验证效果 | J2 | P0 |

### 完整操作路径编写规范

1. **禁止单一动作**：不能仅写"上传剧本文件"或"查看分镜时间轴"。必须写出从触发到目标达成的完整步骤链。
2. **步骤粒度**：每个步骤是用户可感知的单一交互单元（如点击、填写、选择、拖拽），不是系统内部处理。
3. **系统响应显式化**：在操作路径中穿插系统响应，形成"用户动作 ↔ 系统反馈"的闭环。
4. **跨模块跳转**：若操作路径涉及多个模块，必须标明跳转点和数据传递关系（如"在模块 A 完成 X 后，携带数据 Y 进入模块 B"）。
5. **退出与恢复**：路径末尾或关键节点必须说明：用户如何确认完成？中断后如何恢复？

### 反例与正例对比

| 维度 | 反例（不合格） | 正例（合格） |
|------|--------------|-------------|
| **应用场景** | 无 | Given 用户作为新管理员首次登录后台，When 需要配置团队权限... |
| **用户目标** | "so that 我可以导出报表"（Output） | "so that 我能在季度汇报前向 CEO 展示各区域营收对比趋势"（Outcome） |
| **完整操作** | "I want to 导出报表"（1 个动作） | 1. 进入数据中心 → 2. 选择"季度视图" → 3. 勾选"区域"与"营收"维度 → 4. 切换图表类型为"趋势对比" → 5. 确认时间范围 → 6. 点击生成并下载（6 个连续动作） |
| **系统响应** | 无 | 每步操作后系统的实时反馈：加载状态、预览渲染、错误提示、完成通知 |
| **异常处理** | 无 | 若选择的时间范围无数据，系统提示"该季度暂无记录"并提供相邻季度快捷入口 |

### 验收标准（Acceptance Criteria）— Given-When-Then 格式

每个 P0 级用户故事必须附带 **Given-When-Then（GWT）格式**的验收标准，满足：

- **可验证**：每条标准必须能明确判定通过或失败
- **无歧义**：避免"快速"、"友好"、"正确"等主观词，改用具体数值
- **覆盖完整**：每个用户故事至少包含：
  - 1 条 **Happy Path**（正常路径）
  - 1 条 **Negative Path**（错误路径，如权限不足、数据非法）
  - 1 条 **Edge Case**（边界情况，如空数据、极限值、超时）
- **异常状态覆盖（新增）**：若系统状态机定义了异常转移（如进程崩溃、网络中断、产物损坏、系统宕机、审批人离职），关联用户故事必须补充异常场景的 AC：
  - **AC4（异常-进程崩溃）**：Given 节点处于 IN_PROGRESS 且执行进程异常退出，When 系统检测到 exit code 非 0，Then 自动重试 1 次，仍失败则标记 BLOCKED 并通知用户。
  - **AC5（异常-网络中断）**：Given 审批人提交结论时网络中断，When API 超时，Then 保留表单数据，状态保持 GATE_PENDING，提示用户重新提交。
  - **AC6（异常-产物损坏）**：Given 产物 MD5 校验失败，When 开发者查看产物，Then 标记节点 BLOCKED，显示"产物损坏，请重试"，保留历史版本供对比。
- **不预设实现**：写"用户可以导出视频"，而不是"点击导出按钮后调用 FFmpeg"

```markdown
**US-001 — 上传剧本文件**

- **AC1（Happy Path）**：
  Given 用户已登录且处于项目页，
  When 用户选择 .txt / .docx / .pdf 文件（≤ 10MB）并点击上传，
  Then 系统在 5 秒内完成解析并显示角色列表。

- **AC2（Negative Path）**：
  Given 用户已选择文件，
  When 文件格式为 .exe 或大小 > 10MB，
  Then 系统拒绝上传并显示明确错误提示"仅支持 txt/docx/pdf，且不超过 10MB"。

- **AC3（Edge Case）**：
  Given 用户选择了一个 0KB 的空文件，
  When 点击上传，
  Then 系统提示"文件内容为空，请检查后重试"。
```

## 3. 需求追溯矩阵（RTM）

RTM（Requirements Traceability Matrix）是打通用户故事 → 功能需求 → 验收标准的核心工具，确保每个功能需求都有用户故事支撑，每个验收标准都能追溯到具体需求。

### RTM 模板

```markdown
### 需求追溯矩阵（RTM）

| 用户故事 | 功能需求 | 需求描述 | 优先级 | 验收标准 | 状态 |
|----------|----------|----------|--------|----------|------|
| US-001 | REQ-P0-001 | 创建项目时选择开发流程模板 | P0 | AC1: 模板列表加载 < 1s；AC2: 选择后拓扑图预览正确渲染 | Draft |
| US-002 | REQ-P0-003 | SDLC 拓扑图展示 | P0 | AC1: 25 节点正确渲染；AC2: 状态文本与 progress.md 一致 | Draft |
```

### RTM 规则

1. **单向追溯**：每个功能需求必须追溯到一个用户故事。若某功能需求无用户故事支撑，标记为 **伪需求（Pseudo-requirement）**，必须删除或补充用户故事。
2. **双向覆盖**：每个用户故事必须至少映射一个功能需求和一个验收标准。孤立的、无功能需求映射的用户故事视为未落地，标记 WARNING。
3. **编号独立**：用户故事（US-XXX）、功能需求（REQ-P0-XXX）、业务规则（BR-XXX）各自独立编号，禁止混用前缀。
4. **状态同步**：RTM 中的"状态"列与文档头部的"状态"保持一致，冻结时统一更新为 Frozen。

### 伪需求识别 checklist

- [ ] 该功能需求是否解决了某个具体用户的具体痛点？
- [ ] 如果没有该功能，用户是否无法完成其核心目标？
- [ ] 该功能是否有明确的验收标准，且能被用户感知？

若任一答案为"否"，则该需求为伪需求，应从 REQ 清单中移除或降级为 P2。

## 4. 快速检查清单

在输出需求前，逐条确认：

- [ ] 是否所有 P0 需求都有对应的 JTBD 支撑？
- [ ] 是否所有用户故事都关联了 Given-When-Then 验收标准？
- [ ] 每个用户故事的验收标准是否覆盖：Happy Path + Negative Path + Edge Case？
- [ ] 验收标准中是否消除了主观描述（"快速"、"友好"、"正确"）？
- [ ] 需求追溯矩阵（RTM）是否打通：用户故事 ↔ 功能需求 ↔ 验收标准？
- [ ] 是否存在无用户故事支撑的功能需求（伪需求）？
- [ ] 模块名是否已统一为 kebab-case，并与 `feature-XX-{模块}/` 映射一致？
