# PRD-000 概要需求生成器

## 适用场景
- 从零开始规划一个新产品、新系统或大型功能集
- 需要为后续详细需求（PRD-001~PRD-00N）建立不可推翻的基线约束
- 项目启动前明确 Scope、干系人、里程碑和模块边界
- 希望避免 PRD 中出现逻辑矛盾、功能遗漏或技术方案滞后

## 核心职责
1. **四层递进式对话**：严格按 Layer 1→2→3→4 推进，每层评分 < 90 分则阻塞追问
2. **红绿灯评分机制**：每层结束后调用 `references/completeness-scoring.md` 进行量化评分
3. **强制一致性校验**：Layer 4 调用 `references/consistency-checklist.md`，未通过不得输出
4. **基线冻结**：用户确认后，Scope、NFR、核心实体在详细阶段不可推翻
5. **三主题文件输出**：按 OpenSpec 规范输出 3 个主题 Markdown 文件到 `openspec/changes/{变更名}/high-level-requirements/`，将 5 个原子文件按检查视角聚合，减少文件切换
6. **JTBD 框架**：在需求表达中融入 Jobs-to-be-Done 格式，确保从"为什么"出发
7. **多渠道资料收集**：自动调用 `web_search` 收集竞品资料，读取 `@路径` 本地文档
8. **Gate 1 人工冻结提示**：五文件输出完成后，自动宣读 🚪 Gate 1 阻塞提示，等待人工签字后方可进入详细需求阶段

## 使用说明

### 如何触发
- "我要做一个 [产品/系统名称]，先帮我写概要需求"
- "帮我定一下 [项目名] 的需求基线"
- "生成 PRD-000"
- "写 PRD"

### 用户配合要点
1. **准备材料**（可选但推荐）：提前整理业务背景、竞品分析、现有系统资料。
2. **逐层确认**：每层结束后会给出一个"完整性评分"。若标红（<90 分），请根据追问补充信息，不要跳过。
3. **明确说"不"**：当询问 Out-of-Scope 时，请明确列出"系统不需要做什么"。
4. **重视校验阶段**：第四层校验不可跳过，若发现严重问题，会列出清单请您确认。
5. **基线冻结确认**：校验通过并输出完整的三主题文件后，会询问"是否确认基线？"。注意：Step 5.0 的条目回溯矩阵和幻觉检测必须先通过。

## 执行步骤

### Step 0: 初始化（输入分析）

**Gate In — 输入合法性检查**：
1. **Schema 校验**：确认输入包含以下必要元素，缺失则阻塞并提示补充：
   - 变更名称（用于创建目录）
   - 用户需求描述或上游 `brainstorming/` 路径
   - `openspec/config.yaml` 存在且可读
2. **上下文加载**：读取 `openspec/config.yaml` 获取 `artifact_specs.high-level-requirements` 模板和 `required_sections`
3. **来源识别**：区分并记录输入来源类型：
   - `user_direct`（直接输入）、`upstream_skill`（brainstorming 产出）、`external_doc`（`@路径` 本地文档）
4. **来源溯源表**：生成 `input-sources.md`，记录所有输入来源及其关键条目数量
5. **异常拦截**：若上游 `brainstorming/` 目录不存在且用户未提供任何需求描述，立即终止并返回错误："缺少需求输入。请先执行 brainstorming 或提供需求描述。"
6. 检查当前是否已有进行中的变更目录；若无则提示用户先创建变更提案
7. 确认 `openspec/changes/{变更名}/high-level-requirements/` 存在，否则自动创建
8. 读取用户提供的本地资料（`@路径`）和 brainstorming 结果

### Step 1: Layer 1 — 问题界定（Problem Framing）
目标：理解"为什么做"和"为谁做"，收集竞品与技术背景。

1. 按 `references/questioning-guide.md`「第一层提问集」逐题访谈。一次只问一个问题。
2. 在访谈中融入 JTBD 格式："When [场景], I want to [动机], so I can [结果]"。
3. 优先读取 `brainstorming/` 下的 `market-positioning.md`（若存在），提取竞争集合与差异化结论。若不存在，主动调用 `web_search` 搜索行业竞品供用户确认。
4. 输出「问题框架摘要」：业务痛点、目标用户画像、核心价值主张、竞品基准。
5. **替代方案论证（弹性策略）**：在竞品格局后论证"为什么不做其他方案"。
   - **若用户声明"方案已确定"**：只要求对比"现状维持"和"选定方案"两个选项，保留"放弃原因"和"Plan B"两列，记录决策依据即可，无需虚构额外备选。
   - **若用户表示"还没想清楚"**：展开至少 3 个备选方案（含"现状维持"），用表格对比：方案描述、放弃原因、保留价值（Plan B）、决策依据。
   - 核心原则：保留决策思维，减少无意义填写。方案数量服从信息充分度，而非形式要求。
6. 调用 `references/completeness-scoring.md` 进行 Layer 1 评分。
6. 若评分 < 90 分：列出缺失项，阻塞并追问，直到 ≥ 90 分。

### Step 2: Layer 2 — 方案界定（Solution Framing）
目标：确定"系统做什么、不做什么、由哪些模块构成"。

1. 基于 Layer 1 成果，按 `references/questioning-guide.md`「第二层提问集」访谈。
2. 识别系统功能模块，输出 Component Inventory（组件清单），直接映射到后续 `detailed-requirements/feature-XX-{模块}/` 目录。
3. 明确 In-Scope / Out-of-Scope（**Out-of-Scope 必须明确列出**）。
4. **定义核心数据模型（阿里 P8 硬性要求）**：列出所有核心实体，明确每个实体的定义、主数据/事务数据分类、关键属性；用 Mermaid `erDiagram` 绘制实体关系图；说明主数据（Master Data）与事务数据（Transaction Data）的边界；标注关键字段如何驱动 Dashboard 指标计算（数据血缘）。
5. 初步确定技术方案。若存在 `market-positioning.md`，引用其中的 Blue Ocean 差异化结论和战略建议，避免在需求阶段过度预设技术细节。
6. 若用户提供产品 URL，调用 `web_search` / `web_open_url` 进行页面功能审计，补充到模块识别。
7. 调用 `references/completeness-scoring.md` 进行 Layer 2 评分。
8. 若评分 < 90 分：阻塞并追问。

### Step 3: Layer 3 — 成功标准（Success Criteria）与数据需求
目标：量化北极星指标、非功能需求、里程碑、数据需求。

1. 按 `references/questioning-guide.md`「第三层提问集」访谈。
2. 定义量化北极星指标，必须包含：当前基线值、MVP 目标值、测量方法、数据来源、验证方式。
3. 定义全局 NFR（性能、并发、安全、兼容、可维护），附行业对标档位（高/中/低）及可验证的测试场景。
4. 制定里程碑（Phase 1/2/3），确保 Phase 1 为最小可用闭环。
5. **数据需求定义（V3.1 新增）**：
   - 梳理需要采集的用户行为事件（埋点）
   - 定义核心指标的计算逻辑和数据来源（到文件/字段级）
   - 明确数据权限与隔离策略
6. 调用 `references/completeness-scoring.md` 进行 Layer 3 评分。
7. 若评分 < 90 分：阻塞并追问。

### Step 4: Layer 4 — 一致性校验与竞品对标（强制）
目标：捕获内部矛盾、方案缺陷和技术滞后问题。**未通过本层不得输出最终 PRD**。

1. 激活 `references/consistency-checklist.md`：
   - 执行「内部一致性校验」（Scope 自洽性、实体-模块一致性、NFR-技术一致性、角色-权限一致性、**术语行为一致性**、**数据口径与数值一致性**）
   - 执行「竞品对标与技术方案校验」（功能完整性、技术先进性、合规完整性）
2. 读取 `brainstorming/market-positioning.md`（若存在），校验其中的战略建议与假设登记册是否已纳入当前 PRD。若存在矛盾，标记为 🔴 严重问题。若用户未提供竞品技术细节且 `market-positioning.md` 不存在，主动调用 `web_search` 搜索竞品最新功能和技术方案。
3. 问题分级：
   - 🔴 严重：列出问题清单，使用 checklist 中的「用户确认模板」展示，**等待用户明确答复**
   - 🟡 建议：列为「待决策项」
   - 🟢 提示：写入「风险提示」章节
4. **若存在未解决的数据口径矛盾（如不同文件中同一实体数量表述不一致），标记为 🔴 严重问题，禁止输出最终 PRD。**
5. **若存在未解决的 🔴 严重问题，禁止进入 Step 5**。
5. 校验完成后，向用户公示「校验摘要」。

### Step 5.0: 输出前强制检查（Gate Out）

在写入三主题文件前，必须执行以下刚性检查。未通过则禁止输出最终 PRD。

**5.0.1 章节对齐校验**：
1. 读取 `openspec/config.yaml` 当前阶段（`high-level-requirements`）的 `gate_to_next.required_sections` 清单。
2. 读取 `references/system-outline-template.md` 的五文件映射表，确认每个 required_section 的物理归宿文件。
3. 检查待生成的 Markdown 中是否存在对应 H2 标题或明确章节。允许**语义等价映射**（如"核心问题"≈"项目背景与目标"、"边界范围"≈"系统范围（Scope）"），但必须在输出摘要中向用户明示映射关系。
4. 若 required_section 在对应文件中无实质内容（空表、空列表、仅占位符），标记为 🔴 阻塞，禁止进入保存步骤。
5. 若使用了语义等价标题但未精确匹配 config.yaml 的字面要求，标记为 🟡 提示，写入 `00-requirements-overview.md` 的「待决策项」。

**5.0.2 分层自检（快速冲突扫描 + 用户触发式深度回溯）**：

AI 先执行"快速冲突扫描"（限时 30 秒，只标明显矛盾）：
- 对照 Step 0 的 `input-sources.md` 和 Layer 1~3 产出的关键条目，检查是否存在明显矛盾（如同一实体数量不一致、In-Scope 与 Out-of-Scope 重叠、NFR 与里程碑时间冲突）
- 仅标记 🔴 明显冲突，禁止进入保存步骤；无显式冲突即可继续

深度回溯改为用户触发式：
- 向用户展示快速扫描结果："快速冲突扫描已完成，未发现明显矛盾。[需要我执行全量条目回溯吗？]"
- 若用户要求或进入详细阶段前，再执行全量覆盖矩阵（列：来源 ID、关键条目、目标文件、章节、状态、备注）
- 每个 `upstream_skill` 和 `external_doc` 的关键条目必须至少映射到一个输出章节
- 未映射条目在用户触发深度回溯时标记为 BLOCKER；用户明确说"暂不处理"的条目可标记为 WARNING，写入「待决策项」

**5.0.3 幻觉检测与可信度标签**：
- 扫描三主题文件中的所有事实性声明（竞品数据、行业基准、技术参数、用户规模）
- 每条声明必须标注来源：`[来源：{SRC-XXX} 或 {URL} 或 {用户回答}]`
- 无法溯源的声明标记为 `[待验证]`，并统一汇总到 `00-requirements-overview.md` 的「待决策项」
- AI 自行推断但未经验证的内容（如"行业通常采用..."），若无法提供来源，必须改写为假设并登记到假设登记册

### Step 5: 输出与冻结
1. 按 OpenSpec 规范 V3.1 + DocForge 文档规范输出 3 个主题文件。每个文件头部包含标准 YAML Front Matter（见 `references/system-outline-template.md`），所有 `##` / `###` 标题附加 `{#sec-xxx}` 锚点ID。
   - **C4 绑定后置提取**：正文保持自然流畅，禁止在段落首行插入 `@C4-` 标签干扰写作。在文档末尾统一附加《C4 标签映射表》，由 AI 自动从正文提取并填充。用户只需在关键名词首次出现时加粗即可。
   - **单一事实来源索引表（00 开头强制附加）**：00 文档顶部必须放置权威来源索引表，形如：
     ```
     | 主题 | 权威文件 | 00 中的写法 |
     |------|----------|-------------|
     | Non-goals | 00 §3.3 | 一句话摘要 + 链接 |
     | Out-of-Scope | 01 §1.2 | 一句话摘要 + 链接，完整定义见 01 |
     | 业务规则 | 01 §5.1 | 仅引用规则编号 |
     ```
   - `00-requirements-overview.md`（金字塔 + 附录，正文 ≤4 页）：执行摘要、背景与问题（含替代方案论证）、目标与成功指标、范围与边界（一句话/模块级摘要）、用户画像与 JTBD、核心业务流程（Mermaid 旅程图 + 阶段表格）、NFR（只列约束性指标，不写测试场景细节）、核心实体关系（只放 ER 图，不写属性表）、里程碑（只列交付物和时间预期，不写检查清单）、关键风险与假设（Top 3 + Top 3，每条 ≤2 行）、技术约束（4-6 条高层约束，不写框架版本）；附录 A：数据需求详情（指标体系、埋点、属性表，可折叠）；附录 B：发布检查清单（可折叠）；附录 C：运营计划（可折叠）
   - `01-requirements-list.md`：范围与边界（In-Scope / Out-of-Scope / Non-goals）、业务术语表、用户故事与 GWT 验收标准、功能需求清单（REQ-P0-001 顺序编号）、业务规则（含冲突仲裁）、RTM
   - `02-functional-requirements.md`：功能架构图、模块-功能点树状图、端到端用户旅程（含 Happy Path / 替代路径 / 异常退出）、角色职责描述、全局业务规则、状态机（含异常转移）、详细 PRD 清单
   - 各原子章节边界红线独立生效，物理聚合不改变禁止项
2. **单一事实来源原则（新增）**：
   - **Non-goals**：以 `00-requirements-overview.md` 3.3 节（或 4.3 节）为权威来源，`01-requirements-list.md` 1.3 节改为引用："Non-goals 见 00-requirements-overview.md 3.3 节，此处不再重复。"
   - **业务规则**：以 `01-requirements-list.md` 5.1 节为权威来源，`02-functional-requirements.md` 第 5 节改为引用："全局业务规则见 01-requirements-list.md 5.1 节与 5.2 节（含冲突仲裁）。本章仅引用与状态机相关的规则编号。"
   - **Out-of-Scope**：以 `01-requirements-list.md` 1.2 节为权威来源，`00` 文档不重复。
3. 自动保存到 `openspec/changes/{变更名}/high-level-requirements/`
4. 在 `02-functional-requirements.md` 末尾附加「详细需求清单」，列出各模块的映射：
   | 编号 | 模块名称 | 对应目录 | 状态 |
   | DR-001 | {模块A} | `detailed-requirements/feature-01-{模块A}/` | 待编写 |
5. 向用户宣读冻结规则与 Gate 1 阻塞提示，等待人工回复：
   ```text
   ========================================
   Gate 1: 需求冻结 —— 等待人工评审
   ========================================
   产出物已保存至：openspec/changes/{变更名}/high-level-requirements/
   请执行以下操作：
   1. 阅读 3 个主题文件（00-requirements-overview、01-requirements-list、02-functional-requirements）
   2. 重点检查：Scope 是否覆盖业务闭环、P0 需求是否可测试、核心实体是否完整
   3. 如有修改意见，直接编辑文件或在对话中提出，AI 将修正后重新进入 Step 5
   4. 确认无误后，回复"确认冻结"或执行：/skill:human gate=Gate1 action=sign-off
   未获得人工确认前，禁止进入 detailed-requirements 阶段。
   ```
6. 用户确认后，执行基线冻结操作：
   a. 修改三文件 YAML Front Matter 中 `status` 从 `"DRAFT"` 改为 `"FROZEN"`。
   b. 修改 `version` 保持 `"1.0.0"`（BASELINE 不变）。
   c. 更新 `last_updated` 为当前日期（若存在该字段，否则不添加）。
   d. 保存文件后，向用户宣读："PRD-000 基线已冻结，三文件 YAML 元数据已同步更新。后续详细阶段不可推翻 Scope、NFR 与核心实体定义。"
   e. 调用 `self-check` 执行最终一致性校验，输出自查报告。
   f. 更新 `progress-tracker`，标记阶段 2 为"已完成"，阶段 2.5 为"可启动"。按阶段并行规则，阶段 3（概要设计）在 Gate 1 签字后也可同步标记为 IN_PROGRESS，无需等待 Gate 2.5。

## 输出前格式自检

在写入文件系统前，执行 `references/output-checklist.md` 中的格式自检清单。核心检查项：
1. YAML Front Matter 可被标准解析器解析
2. `doc_type` = "PRD"，`c4_binding.level` = "L1"
3. 所有 `##` / `###` 标题含 `{#sec-xxx}` 锚点
4. `c4_binding` 中定义的 actors/external_systems 在正文中有对应的 `@C4-` 标签
5. `dependencies` 中的上游文档 fragment_id 和版本号已填充

任一检查项失败，标记为 🔴 阻塞，禁止保存。

## 输出格式

### 三主题文件结构（OpenSpec 规范 V3.1 — 大厂 PRD 对齐版）

### PRD 边界红线：只写"做什么"，不写"怎么做"

根据 Amazon 6-pager、Google PRD、阿里 P8 等业界规范，PRD-000 必须严格限定为"需求基线"，以下内容为**越界项**，必须拦截或迁移至下游技术设计文档：

| 越界内容 | 应归属文档 | 拦截规则 |
|----------|-----------|----------|
| Component Inventory（前端组件清单） | `detailed-design/LLD-frontend.md` | PRD 只描述用户可见的功能，不描述 UI 组件实现 |
| 页面路由定义（如 `/projects/:id/canvas`） | `detailed-design/LLD-frontend.md` | PRD 描述"用户可访问的页面及其功能"，不写 URL 结构 |
| RBAC 权限矩阵（具体 ACL 表） | `high-level-design/04-quality-attributes.md` 或安全设计文档 | PRD 只写角色职责描述（如"Tech Lead 负责审批 Gate"） |
| 节点状态机代码/样式定义（`classDef page fill:...`） | `detailed-design/LLD-frontend.md` | PRD 中的 Mermaid 只保留业务逻辑抽象 |
| 数据库字段类型、索引、DDL | `detailed-design/db-schema.md` | PRD 只定义核心实体名称和关系，不写字段类型 |
| API 端点规格、请求/响应 Schema | `detailed-design/api-spec.md` | PRD 不写接口契约细节 |
| **具体框架版本**（如 React 19 / FastAPI 0.115） | `detailed-design/LLD-frontend.md` 或 `LLD-backend.md` | PRD 技术约束只写业务层约束（如"执行引擎绑定 Kimi CLI"），不写具体框架选型 |
| **实时通信协议/版本**（如 python-socketio 5 / WebSocket） | `detailed-design/LLD-backend.md` | PRD 不写技术实现层面的通信协议选型 |
| **可观察性工具链**（如 OpenTelemetry + Jaeger） | `detailed-design/observability.md` | PRD 不写监控观测工具的具体选型 |

### 执行摘要（Executive Summary）强制规范

`00-requirements-overview.md` 开头必须包含一页纸执行摘要，让任何干系人能在 **2 分钟内**理解项目价值。格式：

```markdown
## 执行摘要（Executive Summary）

| 项目 | 内容 |
|------|------|
| **产品名称** | {名称} |
| **问题** | {一句话量化痛点，如"研发团队在 AI 辅助开发中面临工具碎片化，导致端到端周期长达 30 天"} |
| **解决方案** | {一句话方案，如"项目级可视化驾驶舱 + AI 编排控制台"} |
| **关键指标** | {北极星指标 + MVP 目标值，如"端到端周期从 30 天降至 14 天"} |
| **目标用户** | {主要 Persona，如"项目经理、开发者、Tech Lead"} |
| **资源需求** | {预期周期/团队规模，如"2-3 个月 / 5 人团队"} |
| **关键风险** | {Top 1 风险及缓解方向} |
```

### 范围与边界强制规范

`01-requirements-list.md` 中除 In-Scope / Out-of-Scope 外，必须增加 **Non-goals（非目标）**：

- **In-Scope**：系统必须做什么（功能域清单）
- **Out-of-Scope**：明确不做的功能（防止需求蔓延）
- **Non-goals**：与目标对称的排除项，解释"为什么本期不做"（如"本期不解决多 Workspace 并发性能，只保证单 Workspace 可用"）

> Non-goals 不等于 Out-of-Scope。Out-of-Scope 是功能列表的否定，Non-goals 是战略层面的主动排除，用于回答"顺手做了吧"的质疑。

### 成功指标（Success Criteria）强制规范

`00-requirements-overview.md` 中的北极星指标和 NFR 必须满足 **SMART + 数据可测量**：

| 维度 | 要求 | 反例 → 正例 |
|------|------|-------------|
| **具体** | 指标定义明确，无歧义 | "响应快" → "页面加载 < 2s（P95）" |
| **可测量** | 定义测量方法和数据来源 | "端到端周期 < 14 天" → "从 Draft → 全部节点通关的耗时，来源：progress.md 的 `start_time` 与 `complete_time` 字段" |
| **有基线** | 提供当前值 | "降至 14 天" → "当前 30 天 → 目标 14 天" |
| **可验证** | 附测试场景或验证方式 | "支持 10 人同时在线" → "并发压测场景：10 用户同时编辑，无冲突提示" |

### 数据需求强制规范

数据需求从 `00-requirements-overview.md` 正文下沉到**附录 A**，但 00 正文必须在"核心实体关系"章节中保留实体名称和关系概述。附录 A 包含（借鉴阿里 P8 模板）：

1. **指标体系**：指标名称、定义、数据来源、计算逻辑、更新频率
2. **埋点需求**：事件名、触发时机、属性字段、用途
3. **数据权限**：数据隔离策略、角色可见范围

> 数据需求是"需要采集什么、算什么、谁可见"的业务契约。概要阶段正文只保留决策结论，字段级细节放在附录。

### 假设、风险与决策日志强制规范

`00-requirements-overview.md` 中的风险与假设改为**决策日志格式**（借鉴 Square/SeatGeek）：

**假设表**：
| 假设 ID | 假设内容 | 置信度（H/M/L） | 若推翻的 Plan B | 关联决策 |
|---------|----------|----------------|-----------------|----------|
| A-001 | {内容} | M | {Plan B 描述} | {关联的架构/产品决策} |

**风险表**：
| 风险 ID | 描述 | 级别 | 缓解措施 | 触发条件 |
|---------|------|------|----------|----------|
| R-001 | {内容} | 高/中/低 | {具体措施} | {何时启动缓解措施} |

> 每个假设必须对应一个风险，每个风险必须有缓解措施和触发条件。禁止只列风险不列缓解策略。

### 核心业务流程（端到端用户旅程）强制规范

`02-functional-requirements.md` 中的核心业务流程必须采用**端到端用户旅程（End-to-End User Journey）**视角编写，禁止仅描述系统内部处理步骤，**禁止包含技术实现细节**。要求如下：

1. **旅程视角**：以目标用户的完整操作路径为主线，从**触发场景**开始，到**用户目标达成**结束，覆盖用户在系统内外的所有关键触点。
2. **阶段划分**：将旅程划分为 4-6 个连续阶段（如：认知 → 准备 → 执行 → 验证 → 完成），每个阶段标明：
   - **用户场景**：用户在什么上下文进入该阶段（时间、地点、前置条件）
   - **用户动作**：用户在该阶段需要执行的完整操作序列（不是单一动作）
   - **系统响应**：系统如何反馈、引导或处理
   - **情绪/痛点**：用户在该阶段可能遇到的焦虑、阻塞或决策困难
   - **出口条件**：用户如何知道自己可以进入下一阶段
3. **流程完整性**：每个主旅程必须同时定义：
   - **Happy Path**：无异常的理想路径
   - **关键替代路径**：用户选择不同分支时的差异旅程（如权限不足时的降级路径）
   - **异常退出**：用户在任意阶段放弃或失败时的系统处理与挽留策略
4. **可视化要求**：必须配合 Mermaid 用户旅程图（`journey`）或端到端流程图（`flowchart TD`）呈现。流程图**只保留业务逻辑抽象**，禁止出现：路由路径、组件名称、样式定义（`classDef`）、状态机代码、`Controller→Service→Repository` 等实现细节。

#### Mermaid 生成约束（用户旅程与业务流程）

生成 Mermaid 图表时，必须遵循 `mermaid-diagrams` skill 的工程化规范规则，并额外满足：

- **用户旅程图（`journey`）**：`section` 命名与文档中的阶段划分严格一致；低评分（1-3）节点必须在图后文字说明痛点原因。
- **端到端流程图（`flowchart`）**：
  - 节点数 > 10 时必须使用 `subgraph` 按阶段分组，组名格式 `subgraph Phase_XXX[阶段名]`
  - 节点 ID 语义化（如 `Phase_Cognize`、`Proc_Upload`），禁止纯中文或纯数字 ID
  - 回流线（如"返回修改"、"重新上传"）必须使用 `-.->` 虚线箭头
  - 平行边合并（如"pass / conditional"指向同一目标时合并为一条边）
  - 样式集中声明：`classDef` 集中在开头，`class` 应用集中在结尾
  - 换行符统一为 `<br>`，禁止 `<br/>`
  - 节点文本禁止直接写 URL 路径、组件名、类名，路由/实现信息用 `%%` 注释承载或不写
  - 形状语义化：决策用菱形 `{}`、页面用矩形 `[]`、成功状态用 `[/.../]`、阻塞状态用 `[\...\]`

### 用户故事与验收标准强制规范

`01-requirements-list.md` 中的用户故事必须超越简单的 `As a... I want to... so that...` 句式，建立与功能需求的严格映射，并配备可验证的验收标准。

#### 用户故事编写规范

1. **应用场景（Context / Given）**：
   - 描述用户进入该场景的前置条件：用户当前在什么页面、已有什么数据、系统处于什么状态
   - 格式建议：`Given [上下文条件], When [触发事件]...`
2. **用户目标（Goal / Outcome）**：
   - 明确用户要达成的**业务结果**（Outcome），而非功能输出（Output）
   - 区分："导出报表"是输出，"在周会前向管理层证明转化率提升"是结果
3. **完整操作（Complete Action Sequence）**：
   - 必须列出用户从触发到目标达成所需的**完整操作步骤链**，禁止只写一个动作
   - 示例："用户从仪表盘选择时间范围 → 点击筛选条件 → 查看实时预览 → 确认导出格式 → 接收下载通知"，而非仅写"用户导出报表"
   - 若涉及跨模块操作，必须标明模块跳转点和数据传递关系

#### 验收标准（Acceptance Criteria）强制规范

每个 P0 级用户故事必须附带 **Given-When-Then（GWT）格式**的验收标准，满足：

- **AC 必须可验证**：每条标准必须能明确判定通过或失败，禁止"快速"、"友好"、"正确"等主观词
- **覆盖完整性**：每个用户故事至少包含：
  - 1 条 **Happy Path**（正常路径）
  - 1 条 **Negative Path**（错误路径，如权限不足、数据非法）
  - 1 条 **Edge Case**（边界情况，如空数据、极限值、超时）
  - 若涉及性能/安全，补充 **Non-behavioral AC**（如"响应时间 < 200ms（P95）"）
- **GWT 格式示例**：
  ```
  AC1: Given 用户已登录且角色为 Admin/PM，When 点击"新建项目"并填写有效名称、描述后提交，Then 系统在 500ms 内创建项目并自动跳转 Dashboard。
  AC2: Given 项目列表存在 N 个项目，When 用户点击表头"创建时间"，Then 列表按该字段升序/降序切换排列。
  ```

#### 需求追溯矩阵（RTM）强制规范

`01-requirements-list.md` 末尾必须包含**需求追溯矩阵（Requirements Traceability Matrix）**，打通用户故事 → 功能需求 → 验收标准：

```markdown
### 需求追溯矩阵（RTM）

| 用户故事 | 功能需求 | 需求描述 | 优先级 | 验收标准 | 状态 |
|----------|----------|----------|--------|----------|------|
| US-001 | REQ-P0-001 | 创建项目时选择开发流程模板 | P0 | AC1: 模板列表加载 < 1s；AC2: 选择后拓扑图预览正确渲染 | 待评审 |
| US-002 | REQ-P0-003 | SDLC 拓扑图展示 | P0 | AC1: 25 节点正确渲染；AC2: 状态图标与 progress.md 一致 | 待评审 |
```

> 规则：每个功能需求必须追溯到一个用户故事，否则视为伪需求。每个用户故事必须至少映射一个功能需求和一个验收标准。

#### 编号体系统一规范

采用单维度分层编号，禁止混用后缀（如 `REQ-P0-001a`）：

```
REQ-P0-001    ← 需求（Requirement），独立顺序编号
    AC-001    ← 验收标准（Acceptance Criteria），从属于 REQ
US-001        ← 用户故事（User Story），独立顺序编号
BR-001        ← 业务规则（Business Rule），独立顺序编号
DR-001        ← 详细需求（Detailed Requirement），在详细设计阶段产出
```

- 禁止同一维度内使用 a/b/c 后缀区分子项（如 `REQ-P0-001a`），应直接升编号
- 禁止跨维度混用前缀（如用户故事不能用 `REQ-` 开头）

#### 状态标识规范

正式 PRD 中**禁止**使用 Emoji（🚀🟡🔴⏸⬜🟢）作为状态标识。统一使用标准文本标签：

| 场景 | 禁用 | 启用 |
|------|------|------|
| 文档状态 | 🟡草稿 / 🟢已冻结 | Draft / Frozen / Reviewing |
| 需求状态 | ⬜待办 / 🟢已完成 | NOT_STARTED / IN_PROGRESS / COMPLETED |
| 审批状态 | ⏸等待 / 🔴阻塞 | PENDING / BLOCKED / APPROVED |
| 流程节点 | 🟡进行中 / 🟢已通关 | IN_PROGRESS / PASSED / FAILED |

> Emoji 仅在内部沟通或非正式草稿中使用，正式交付物必须使用文本标签，确保 PDF 导出、打印、代码解析时的兼容性。

#### Skill 输出 Emoji 禁令（新增）

- **Skill 在生成正式 PRD 文件时**，不得在 Markdown 内容中使用任何 Emoji（包括 🚀、🟡、🔴、⏸、⬜、🟢、🚪 等）。
- **仅允许**在以下场景使用 Emoji：
  - 向用户展示的交互提示（如"本层评分为 XX 分 🟢"）
  - 非正式草稿或内部评论
- 正式交付物中的状态必须使用文本标签：`Draft / Frozen / NOT_STARTED / IN_PROGRESS / COMPLETED / BLOCKED / GATE_PENDING / PENDING / APPROVED`

#### 故事地图位置

在 `02-functional-requirements.md` 的模块-功能点树状图中，必须标明每个用户故事在端到端旅程中的所属阶段，形成"用户故事地图（User Story Mapping）"的雏形。

**边界红线（每章独立生效）**：
- 需求清单：禁止写技术方案、实现细节、字段类型；用户故事禁止只写单一动作而无上下文和完整路径
- 功能结构：禁止写 UI 像素级设计、数据库表结构
- 业务规则：禁止写代码逻辑、算法流程、API 端点；用户旅程禁止写成系统模块调用链

#### 功能点粒度校验（模块-功能点树状图）

`02-functional-requirements.md` 中的每个功能点必须满足以下测试，任一不满足则降级为"界面元素"并从本表移除：

> **功能点粒度四问**：1) 能否用一句话描述用户完成的**完整任务**（从触发到结果）？2) 能否对应一个独立的用户故事标题？3) 是否产生**可感知的业务结果**（而非仅系统内部状态变更）？4) 若删除该功能点，用户是否**无法完成某个核心目标**？

| 层级 | 正确粒度 | 错误示例 | 正确示例 |
|------|----------|----------|----------|
| 模块 | 一个业务域，用户能说出"我在管理 XX" | "项目初始化页面"、"SDLC 画布组件" | "项目生命周期管理"、"SDLC 编排与执行"、"审批与质量门控" |
| 功能点 | 一个完整用户任务，从开始到结束产生可感知结果 | "点击创建项目按钮"、"选择模板下拉框"、"提交审批表单" | "基于模板创建新项目并初始化工作流"、"在画布中执行 Skill 并查看产物状态"、"提交 Gate 审批并获取决策结论" |

### 优先级标记
- **P0**：必须交付，缺一则系统不可用
- **P1**：重要，影响核心体验
- **P2**：优化项，可延后

### 模块命名约束
`02-functional-requirements.md` 中的模块名必须与 `detailed-requirements/feature-XX-{模块名}/` 目录名保持一致（kebab-case），直接影响 `detailed-requirements` Skill 的目录拆分。

## Gotchas

### 分层与校验纪律
- **不要一次问完所有问题**。必须分层进行，每层确认完整性后再进入下一层。
- **第四层校验不可跳过**。即使前三层评分都是绿灯，也必须在输出前执行一致性校验。
- **Step 5.0（Gate Out）不可跳过**。章节对齐、条目回溯矩阵、幻觉检测未全部通过前，禁止写入最终文件。
- **发现严重问题时禁止自动修改**。必须列出问题清单，等待用户确认后再修正 PRD。
- **Gate 1 是阻塞性闸门，不可跳过**。即使三主题文件内容看似完整，也必须获得人工签字确认后才能冻结基线。AI 不得因"看起来没问题"而自动确认。
- **基线冻结后变更需升版**。若冻结后发现 Scope 遗漏或 NFR 错误，必须通过 `human` Skill 记录变更决策，升版至 v1.1+ 并通知所有下游 Skill 重新校验。

### 输入覆盖与幻觉
- **上游条目遗漏是 BLOCKER**：若 `brainstorming/requirement-draft.md` 中明确登记的业务规则、数据约束未进入 PRD，必须标记为 🔴 阻塞，禁止输出
- **禁止未标注来源的事实陈述**："行业平均转化率 3%"这类数据若无法提供来源，必须改写为假设（"假设行业平均转化率为 3%，置信度 M，待验证"）
- **外部文档（`@路径`）必须逐条回溯**：用户提供的本地资料不是"参考背景"，而是需求来源。每个被用户高亮或明确提及的条款都必须在覆盖矩阵中有对应映射

### 范围与边界
- **Out-of-Scope、In-Scope、Non-goals 三者缺一不可**。Out-of-Scope 是功能否定，Non-goals 是战略排除。缺少 Non-goals 会导致开发过程中不断有人提出"顺手做了吧"。
- **核心实体一旦定义，不可在详细 PRD 中擅自新增**。如果详细阶段发现需要新实体，必须回溯修改 PRD-000 并升版。
- **模块名一旦确定，详细阶段不可改**。修改模块名会导致 `detailed-requirements/feature-XX-{模块}/` 目录重命名，影响所有下游 Skill。

### 需求质量
- **P0 需求必须可测试**。每个 P0 需求应对应至少一个用户故事的验收标准，且验收标准必须使用 Given-When-Then 格式。笼统描述如"可创建项目，指定名称、描述"视为不合格。
- **用户故事与功能需求必须建立映射**。无用户故事支撑的功能需求视为伪需求，必须在 RTM 中显式标注。每个功能需求必须追溯到一个用户故事，否则标记 BLOCKER。
- **用户故事必须描述完整操作路径**。禁止只写"用户可以导出报表"这类孤立动作，必须补充：用户在哪触发（Given）、经过哪些步骤（When...Then...）、最终获得什么业务结果。缺少完整操作链的用户故事视为 WARNING。
- **核心业务流程禁止写成系统模块调用链**。如"前端调用 API → 后端校验 → 写入数据库"是错误写法；正确写法是"用户提交表单 → 系统实时校验并提示错误 → 用户修正后再次提交 → 收到成功通知"。

### 指标与数据
- **成功指标必须可测量**。只写"端到端周期 < 14 天"是不够的，必须补充：测量方法（从哪个状态到哪个状态）、数据来源（哪个文件的哪个字段）、当前基线（30 天是怎么测出来的）。
- **数据需求不是可选项**。作为"可视化驾驶舱"或数据驱动产品，PRD 必须包含指标体系、埋点事件和数据权限章节，否则前后端联调时会反复确认字段。
- **假设与风险必须与决策挂钩**。每个假设必须有置信度（H/M/L）和推翻后的 Plan B；每个风险必须有缓解措施和触发条件。禁止只列假设不列风险，或只列风险不列缓解策略。

### 表达规范
- **术语表必须前置**。禁止将术语表放在文档第 4 节以后。应在文档第 2 节（或紧随执行摘要之后）放置术语表，或在术语首次出现时随文注释。
- **Emoji 禁止作为正式状态标识**。正式 PRD 中使用标准文本标签（Draft / Active / BLOCKED / PASSED），Emoji 仅在内部草稿中使用。
- **编号体系禁止混用后缀**。取消 `REQ-P0-001a` 这种后缀，改为 `REQ-P0-001`、`REQ-P0-002` 顺序编号。用户故事、业务规则、详细需求各自独立编号，禁止前缀混用。

### PRD 边界（只写"做什么"，不写"怎么做"）
- **Component Inventory 不应出现在 PRD**。组件清单（C-001 ~ C-027）属于前端技术设计文档（LLD），PRD 只描述用户可见的功能。
- **页面路由定义不应出现在 PRD**。如 `/projects/:id/canvas` 属于前端架构师职责，PRD 只描述"用户可访问的页面及其功能"。
- **RBAC 权限矩阵过于工程化**。PRD 只写角色职责描述（如"Tech Lead 负责审批 Gate，无权创建项目"），具体 ACL 表放到安全设计文档。
- **Mermaid 流程图只保留业务逻辑**。禁止在 PRD 的流程图中写路由路径、组件名、样式定义（`classDef`）、状态机代码。流程图节点应以业务动作命名。
- **用户旅程与模块树必须互锁**。若 `02-functional-requirements.md` 中的端到端旅程某阶段在模块树中找不到对应功能点，或模块树中的功能点在旅程中从未被用户触发，标记为 BLOCKER。

## 示例

**输入**："我们要做一个 AI 短剧生成平台，让用户能从剧本直接生成视频。"

**Layer 1**：痛点（制作周期长 2-3 个月）、目标用户（独立制片人）、价值主张（剧本到成片从天级）、竞品（Runway、剪映）
**Layer 2~4**：模块（剧本工坊、渲染中心）、Out-of-Scope（视频分发、支付）、NFR（10人并发）、🔴 严重——Out-of-Scope 含"支付系统"但 In-Scope 未明确付费路径，用户补充"Phase 1 免费+试用"后解除。
