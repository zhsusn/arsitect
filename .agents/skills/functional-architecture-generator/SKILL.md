# 系统功能架构文档生成器

基于 PRD 或需求文档，识别业务域边界，将零散功能点归纳为高内聚、低耦合的系统模块，并输出标准化的《系统功能架构及模块划分》文档。

## 适用场景
- 从概要需求/详细需求推导系统模块结构
- 梳理业务域边界与功能归属
- 生成可渲染的 Mermaid 功能架构图
- 输出模块清单与关键架构说明

## 分区划分方法论（自行推导，禁止套用固定模板）

根据需求内容，从以下维度中选择 **1–2 个最贴合的划分视角** 进行分区：

| 划分视角 | 适用场景 | 示例 |
|---------|---------|------|
| **用户角色视角** | 不同角色使用完全不同的功能集合 | 用户端、运营端、管理端、商家端 |
| **业务域视角** | 业务天然存在独立闭环 | 商品域、订单域、支付域、物流域 |
| **系统层级视角** | 存在明显的上下游或分层关系 | 接入层、业务层、核心服务层、数据层 |
| **内外部边界视角** | 大量功能依赖外部系统 | 自营系统、第三方服务、外部监管 |
| **生命周期视角** | 业务有明显的阶段递进 | 贷前、贷中、贷后；售前、售中、售后 |

**分区原则：**
1. **需求驱动**：分几区、叫什么名，完全由需求内容决定，禁止强行凑成固定数量
2. **互斥且穷尽**：每个功能模块只能属于一个分区；所有功能必须被覆盖
3. **命名即意图**：分区名要能一眼看出该区的业务性质（如"智能风控域"优于"系统A"）
4. **数量控制**：通常 3–6 个分区为宜；若超过 6 个，考虑合并弱相关域或升级为"一级域→二级子域"两级结构

## 布局推荐规范（按需调整，非强制）

### 布局决策树
1. 若系统有明显的**分层结构**（接入/业务/数据）→ 使用 `flowchart TB`
2. 若系统为**扁平微服务**（Gateway → 多个平级服务）→ 使用 `flowchart LR`
3. 若单个业务域模块数 ≤ 4 → 使用 `direction TB` 单列
4. 若单个业务域模块数 > 4 → 使用 `direction LR` 双列，或拆分为二级 subgraph

### 通用布局规则
- **分区内部**：推荐在 `subgraph` 内使用 `direction TB` 或 `direction LR`，根据模块数量灵活选择
- **外部依赖域布局**：
  - 若外部依赖模块数 **≤ 4**：使用单列 `direction TB` 纵向堆叠，置于最下方
  - 若外部依赖模块数 **> 4**：使用**多行网格**，每行一个 `subgraph`（`direction LR`），每行最多 **4 个**模块
- **层级关系**：若存在明显的上下游，用实线箭头 `-->` 体现流向；外部依赖用虚线箭头 `-.->` 指向被依赖的内部模块
- **节点语法**：统一使用 `ID[文本]:::样式` 或 `ID[文本]`。无特殊字符时无需引号；若节点文本含 `{}` / `[]` / `/` 或中文双引号，必须外层加双引号包裹整个节点（如 `ID["跳转 /{id}"]`），遵循 `mermaid-diagrams` 跨平台兼容性规则

> **排版技巧（可选）**：若需在 subgraph 内实现严格的两列布局，可嵌套两个 `direction TB` 的子 subgraph。  
> 注意：此技巧在 Mermaid v10+ 中稳定，在部分旧版渲染器（如某些 CI 环境）中可能出现方向异常。

## 颜色编码策略（层次区分 + 同色系渐变）

核心原则：大域（subgraph）背景色与内部节点色必须属于同一色系，但色阶不同，形成"大色块区分层次、小色块标识模块"的视觉效果。

**色阶规范：**
- 大域背景（层次色）：使用同色系 100–200 色阶（如红 `#ffcdd2`、蓝 `#bbdefb`），填充整个 subgraph 区域
- 节点填充（模块色）：使用同色系 50 色阶（如红 `#ffebee`、蓝 `#e3f2fd`），比背景更浅
- 描边与文字：使用同色系 700–900 色阶（如红 `#c62828`、蓝 `#1565c0`），确保对比度

**常见分区颜色参考表：**

| 分区类型 | 大域背景（100/200） | 节点填充（50） | 描边/文字（700/900） |
|---------|-------------------|--------------|-------------------|
| 前台系统 / 用户端 / 客户端 | `#ffcdd2` | `#ffebee` | `#c62828` |
| 管理平台 / 运营端 / 管理端 | `#ffe0b2` | `#fff3e0` | `#ef6c00` |
| 业务中心 / 核心服务 / 业务中台 | `#fff9c4` | `#fffde7` | `#f9a825` |
| 大数据平台 / 数据层 / 智能层 | `#c8e6c9` | `#e8f5e9` | `#2e7d32` |
| 基础办公 / 通用支撑 / 公共服务 | `#cfd8dc` | `#eceff1` | `#455a64` |
| 三方平台 / 外部依赖 / 外部服务 | `#bbdefb` | `#e3f2fd` | `#1565c0` |

**分配规则：**
- 优先匹配：若推导出的分区名与上表某类型语义一致，直接使用对应色系
- 无匹配时自选：从以下扩展色系中选取，保证同文档内颜色不重复、相邻分区色差明显
- 扩展色系：浅绿 `#dcedc8`/`#f1f8e9`/`#33691e`、浅棕 `#d7ccc8`/`#efebe9`/`#4e342e`、浅粉 `#f8bbd0`/`#fce4ec`/`#880e4f`

**Mermaid 实现方式：**
- 节点样式：`classDef` 定义（如 `classDef channel fill:#ffebee,stroke:#c62828,stroke-width:2px,color:#c62828`）
- 大域背景：`style P1 fill:#ffcdd2,stroke:#c62828,stroke-width:3px` 对每个 subgraph 单独设置

## 输出格式

### 1. 架构划分说明
- **划分视角**：说明本次采用了哪种视角（如"按用户角色 + 业务域"）
- **分区列表**：列出每个分区的名称、定位一句话、包含模块数量、分配的色系

### 2. 系统架构总览（Mermaid 代码）
遵循本文的布局推荐规范与颜色策略，输出可直接渲染的完整 Mermaid 代码。必须包含：
- 顶部 `classDef` 样式定义
- 每个大区的 `subgraph` 嵌套结构
- 每个大区的 `style` 背景色设置
- 层级流向箭头（实线 + 虚线）

### 3. 模块清单表
| 分区 | 模块名称 | 功能职责（一句话） | 对应需求章节 | 依赖/被依赖 |
|------|---------|------------------|------------|------------|
| ... | ... | ... | ... | ... |

### 4. 关键架构说明
- **核心引擎/中枢模块**：指出哪些模块承担规则、决策、调度、编排职责，说明其输入输出
- **跨分区交互**：描述 2–3 条最典型的跨区数据流或调用链
- **外部依赖**：若存在三方/外部系统，说明对接内容（传什么数据、调什么能力）

## 交付与校验

### 1. 语法校验（必填）
生成 `.mmd` 代码后，必须先验证语法正确性，再嵌入文档或交付用户。

**兼容性预检（在运行 mmdc 前执行）：**
1. 所有 `[/` 都有对应的 `/]`，所有 `[\` 都有对应的 `\]`
2. 节点文本含 `{}` / `[]` / `/` 的已加双引号包裹
3. 箭头标签 `|...|` 中含 `/` /空格/状态码的已加双引号
4. 节点文本无中文/英文内部引号
5. 未使用 HTML 实体（`&#123;` 等）

> 详细规则见 `mermaid-diagrams` skill 的「跨平台兼容性规则」章节。

```bash
# 本地校验（推荐）
mmdc -i architecture.mmd -o /tmp/test.png 2>&1

# 无 mmdc 时使用 Kroki API
curl -s -X POST -H "Content-Type: text/plain" \
  --data-binary @architecture.mmd \
  https://kroki.io/mermaid/svg -o /tmp/test.svg
```

### 2. 交付形式
- **默认交付**：将 Mermaid 代码块嵌入 Markdown 文档（GitHub/GitLab/IDE 原生支持渲染）
- **需独立图片时**：使用 `mermaid-diagrams` skill 的工具链导出 PNG/SVG/PDF
  - `mmdc -i architecture.mmd -o architecture.png -w 2048 --backgroundColor white`
  - 或调用 Kroki API 生成

### 3. 渲染异常回退
若嵌套 subgraph 出现方向异常或样式丢失：
- 检查 Mermaid 渲染器版本（v10+ 最稳定）
- 简化布局：将多级嵌套 subgraph 降级为单层 subgraph + 节点分组
- 移除复杂的 `direction LR/TB` 嵌套，改用默认流向

## Gotchas / 常见陷阱

- **不虚构模块**：不得生成需求文档中不存在的技术组件或业务模块
- **粒度控制**：单个分区内的模块数量控制在 4–8 个；超过 8 个需先合并为"子系统"，再在下级展开
- **可验证性**：每个模块必须能在详细需求中找到至少一处原文支撑；建议在模块清单表的"对应需求章节"列中标注
- **命名规范**：模块名采用"业务对象 + 动作/能力"或纯业务名词，禁止出现"模块1"、"功能A"、"系统X"等无意义命名
- **Mermaid 渲染兼容性**：嵌套 subgraph 方向在 Mermaid v8.x 和部分 CI 渲染器中可能异常，建议在本地渲染确认后再提交
- **跨平台语法陷阱**：含 `{}` 的节点未加引号、梯形斜杠不成对 `[/text]`、箭头标签 `|4xx/5xx|` 未引号化，都会导致 Obsidian / 旧版 Mermaid 整段白屏。生成后必须跑 `mermaid-diagrams` 兼容性检查清单
- **颜色对比度**：避免将深色文字放在深色背景上，确保打印（黑白）时仍能区分区域边界
- **零遗漏**：详细需求中提到的每一个功能点，必须映射到具体模块；如有无法归类的，单独列出"待归类功能"并说明原因
