# 规范化提交信息生成器

## 适用场景
- 代码提交和变更归档

## 输入
- git diff
- 修改文件列表
- 变更目的

## 工作流程
1. 判断变更类型。
2. 推断作用域。
3. 生成 72 字以内首行。
4. 必要时补正文。
5. 如变更不相关，建议拆提交。

## 输出格式
- commit message
- 可选正文
- 拆分建议

## Gotchas
- **单一变更**：一个提交对应一个逻辑变更。若 diff 中同时包含功能新增和 Bug 修复，必须建议拆分为两个 commit，不要把不相关变更塞进同一句话。
- **类型隔离**：不把功能和修复混在一起。`feat` 和 `fix` 不得出现在同一个提交中，若边界模糊（如"修复某功能以支持新场景"），按"主要意图"分类并标注。
- **保守描述**：不知道作用域时用更保守描述。若无法确定变更影响的具体模块，使用通用作用域（如 `core:`）或省略作用域，不要凭空编造模块名。
