# Self-Check

产出物自查引擎。在每个阶段完成后对产出物进行系统性自查，确保质量门控通过后方可进入下一阶段。

## 适用场景

- 阶段完成后强制门控（概要需求 → 详细需求 → 设计 → 编码 → 测试 → UAT → 发布 → 收尾）
- 用户主动要求检查当前文档/代码/交互规格的完整性
- 发现文档间可能存在矛盾或引用失效时
- 编码完成后验证接口与 API 契约一致性
- 测试阶段验证覆盖率与测试有效性
- UAT 阶段验证 uat-report.md 质量与用户故事覆盖度（V2.1 新增）

## 核心原则

1. **零信任原则**：不假设产出物正确，每个结论必须有证据
2. **证据溯源**：每个检查结果必须标注证据位置（文件:行号 或 章节）
3. **原子化验证**：将复合声明拆分为独立可验证单元
4. **测试必须能失败**：覆盖率检查不仅看数字，还要看测试是否真正验证行为

#### 内容一致性

对比当前产出物与上游文档，提取上游关键结论并逐项核对：
1. 在上游文档中识别所有事实性声明（如"系统面向 C 端用户"、"响应时间 < 2s"）
2. 在当前产出物中查找对应表述
3. 检查当前产出物引用的上游文档版本号是否为最新（如 `spec.md` 声明基于 PRD-000 v1.2，而当前 PRD-000 已升版至 v1.3）
4. **数据口径与数值一致性（新增）**：扫描所有涉及实体数量、节点数量、用户数量、容量上限的数值描述，检查全文是否保持唯一口径；同一数值在不同文件中的表述是否完全一致
5. **单一事实来源检查（新增）**：检查同一事实（如 Non-goals、Out-of-Scope、业务规则）是否只在一个权威文件中定义；其他文件是否通过引用指向权威来源，而非段落级重复定义
6. 标记状态：一致 / 矛盾 / 缺失 / 版本断裂 / 数据口径冲突 / 重复定义 / 需澄清

#### 内容完整性

对照 `config.yaml` 的 `required_sections` 检查章节完整性：
1. 读取该阶段的 `required_sections` 清单
2. 扫描产出物 Markdown 标题结构（H1/H2）
3. 检查每个 required_section 是否有对应内容（非空）
4. 检查 Mermaid 图表是否存在且语法正确

#### 交叉引用有效性

验证文档间 `@引用` 和链接的有效性：
1. 正则扫描 `@路径` 引用
2. 验证文件是否存在（支持相对路径和项目根目录解析）
3. 验证锚点（`#标题`）是否存在
4. 验证 Markdown 标准链接 `[text](path)`

#### 无内部矛盾

检测同一文档内的逻辑冲突：
1. 提取文档中所有约束、规则、数值
2. 检测同一术语的不同定义
3. 检测数值范围矛盾与数据口径冲突（如 A 文件写"25 个节点"，B 文件写"41 个 Skill"，未明确统计口径）
4. 检测跨文件重复定义（如 00 和 01 同时定义 Non-goals 或 Out-of-Scope，未标注权威来源）
5. 检测状态机转换冲突
6. 检测权限矩阵矛盾

#### 变更影响分析（按需）

当用户明确提出"修改某功能/需求/设计，会影响哪些模块/文档"时触发：
1. 定位变更涉及的上游需求 ID 或设计章节
2. 遍历当前变更目录下的所有下游文档（`high-level-requirements/`、`detailed-requirements/`、`high-level-design/`、`detailed-design/`、`plan.md`、`tasks.md`）
3. 识别直接关联：继承该需求/设计的下游文档
4. 识别间接关联：接口耦合、数据依赖、状态机引用、模块间字段共用
5. 输出受影响清单，按影响程度分级：
   - 🔴 **必须同步修改**：直接继承或依赖该变更的文档
   - 🟡 **建议评审**：间接引用或接口耦合的文档
   - 🟢 **无影响**：不在该变更链路上

> **继承自原 `prd-trace-matrix`**：本分析模式替代了独立的 REQ-TRACE 矩阵维护，将集中式追溯转化为按需分析，降低维护成本。

#### 接口一致性（编码阶段）

对比代码中的接口与 `api-spec.md` / `openapi.yaml`：
1. 提取代码中的 API 定义（后端：路由装饰器；前端：API 调用函数）
2. 提取 spec 中的接口表格 / OpenAPI paths
3. 对比路径、HTTP 方法、请求参数、响应结构、异常码
4. 检查异常处理覆盖：设计中的异常场景是否都有代码处理

#### 覆盖率与测试有效性（测试阶段）

1. 覆盖率门控：运行 `pytest --cov`，检查总体覆盖率 ≥ 70%
2. 反模式检测（Green Mirage Audit）：
   - 空断言（`assert True`、`assert None is None`）
   - 同义反复（`assert len(list) == len(list)`）
   - 过度 mock（mock 了被测对象本身的核心逻辑）
   - 检查实现而非行为（直接断言内部状态）
   - 无法失败的测试（无 assert 或无条件通过）
3. 边界条件覆盖：空输入、最大值、异常值、并发场景

4. **运行时行为验证覆盖度（V3.2 新增）**：
   扫描 `tests/unit/` 和 `tests/integration/` 目录，检查是否包含运行时行为验证：
   - 每个含 I/O 边界（DB/网络/文件/缓存）的模块，是否至少包含 Baseline + Edge + Fault 三类探测？
   - 是否使用 Spy/Mock 验证外部依赖调用次数和参数？
   - 是否包含 Pre-state / Post-state 状态快照断言？
   - 异常路径测试是否经过运行时验证（资源回滚、无脏数据）？
   
   评分标准：
   - 含 I/O 边界的模块缺少 Baseline 探测 → 🔴 BLOCKER
   - 含 I/O 边界的模块缺少 Edge 或 Fault 探测 → 🟡 WARNING
   - 无 Pre/Post 状态快照断言 → 🟡 WARNING
   - 异常路径仅断言异常类型、未验证资源回滚 → 🟡 WARNING
   
   - 运行时测试文件是否包含标准头模板（Matrix / Boundaries / Gates / Cleanup）？
   - 运行时测试是否使用环境隔离（DB 事务包裹、临时目录、Mock 重置）？
   - 属性测试是否设置了性能预算（CI 中 `max_examples <= 50`）？
   - 是否禁止直接探测生产环境（Destination Gate 通过）？
   
   新增评分标准：
   - 运行时测试文件缺少标准头模板 → 🟡 WARNING
   - 无环境隔离（如直接操作真实 DB、未使用临时目录） → 🔴 BLOCKER
   - 属性测试无性能预算控制 → 🟡 WARNING
   - 探测目标为生产环境或未确认的真实端点 → 🔴 BLOCKER

5. **属性测试覆盖度（V3.2 新增）**：
   检查是否包含 Property-Based Testing：
   - 序列化/反序列化模块是否包含 Roundtrip 属性测试？
   - 状态变更模块是否包含 Idempotence 属性测试？
   - 纯函数/算法模块是否包含 Invariant 属性测试？
   
   评分标准：
   - 适合 PBT 的模块完全缺失属性测试 → 🟡 WARNING
   - 属性测试使用随机数据生成工具（hypothesis/fast-check/jqwik）→ ✅ 加分项

6. **覆盖率盲区填补验证（V3.2 新增）**：
   检查覆盖率报告中的盲区是否已被运行时验证填补：
   - 覆盖率报告中的未覆盖分支，是否有对应的运行时行为验证测试？
   - 静态测试覆盖但从未实际运行的分支，是否有针对性的输入数据测试？
   
   评分标准：
   - 未覆盖分支无运行时验证填补 → 🟡 WARNING
   - 盲区填补后重新运行覆盖率仍低于 70% → 🔴 BLOCKER
   
   - 运行时测试是否处理了非确定性因素？
     - 涉及随机数、时间戳、并发 ID 的探测是否固定了随机种子？
     - 是否使用语义等价断言替代精确值断言？
   
   新增评分标准：
   - 运行时测试因非确定性因素导致 flaky（不稳定通过）→ 🔴 BLOCKER
   - 未使用语义等价断言（如直接断言 `timestamp == 1699123456`）→ 🟡 WARNING

#### 交互规格完整性（详细需求阶段，V2.1 新增）

检查 `interaction-spec.md` 中按钮级交互状态机的覆盖度：
1. **元素覆盖度**：每个页面是否列出所有可交互元素（按钮、输入框、下拉框、链接、开关）
2. **状态机完整性**：每个交互元素是否包含触发方式 / 前置条件 / 立即反馈 / 成功结果 / 失败结果 / 异常分支
3. **异常分支覆盖**：网络中断、权限不足、数据为空、超时是否均有定义
4. **埋点完整性**：关键操作（提交、支付、删除、状态切换）是否均有埋点事件定义
5. **页面跳转一致性**：`interaction-spec.md` 中的页面跳转图是否与 `prototype.md` 中的页面结构一致
6. **与 logic.md 的一致性**：交互元素触发的业务规则是否与 `logic.md` 中的状态机定义兼容

评分标准：
- 元素覆盖度 < 90% → 🔴 BLOCKER
- 核心流程（P0 用户故事涉及页面）存在无状态机定义的交互元素 → 🔴 BLOCKER
- 异常分支缺失 > 2 处 → 🟡 WARNING
- 埋点缺失 > 3 处 → 🟡 WARNING

#### 设计文档质量检查（概要设计阶段，V2.2→V3.0 更新）

检查 `design/` 目录下 6 个主题文件的规范性与可追溯性：
1. **需求可追溯性**：每个 PRD 需求（REQ-XXX）在 `01-05` 主题文件中至少有一个章节回应；`00-design-overview.md` 也要有追溯段落
2. **数据模型完整性**：`02-data-flow.md` §1 是否含逻辑实体、类型、约束、核心表清单（禁止字段级细节，但逻辑约束必须明确）
3. **API 契约完整性**：`02-data-flow.md` §2 是否完整定义模块间通信模式、数据契约、版本策略
4. **错误场景文档化**：`03-runtime-behavior.md` §3 是否覆盖业务/系统/网络/AI 模型错误分类，及与 `05-ops-governance.md` 回滚章节的联动
5. **安全考量覆盖**：`04-quality-attributes.md` §1 是否覆盖认证授权、数据加密、网络隔离
6. **性能策略对应**：`04-quality-attributes.md` §2 是否对 `05-non-functional.md` 中的性能需求有明确策略（缓存、异步、容量）
7. **依赖版本标注**：`00-design-overview.md` §2.3 中的外部依赖是否标注版本或版本范围
8. **架构策略对比**：`01-architecture-core.md` §2.2 是否包含备选方案对比矩阵（至少 2 个备选）及 ADR 格式决策记录
9. **目录-架构一致性**：`01-architecture-core.md` §3 的目录层级是否与 §1 的架构分层一一对应
10. **回滚可操作性**：`05-ops-governance.md` §2 的回滚步骤是否明确到"执行某脚本→验证→切换流量"级别

评分标准：
- 需求可追溯性缺失 ≥ 3 处 → 🔴 BLOCKER
- `01-architecture-core.md` 无备选方案对比 → 🔴 BLOCKER
- `03-runtime-behavior.md` 未与 `05-ops-governance.md` 回滚章节联动 → 🟡 WARNING
- 目录-架构不一致 → 🟡 WARNING
- 其他单项缺失 → 🟡 WARNING

#### 跨文件一致性检查（概要设计阶段，V3.0 新增）

检查 6 个主题文件之间的交叉一致性，生成 `self-check-report.md`：
1. **技术栈覆盖度**：`01-architecture-core.md` §2 中的存储组件选型是否覆盖 `02-data-flow.md` §1 中的全部存储需求
2. **架构-目录一致性**：`01-architecture-core.md` §1 的架构分层与 §3 的目录层级是否一一对应
3. **状态机-模块职责兼容性**：`03-runtime-behavior.md` §1 中的全局状态是否在 `02-data-flow.md` §3 的模块职责中有对应处理方
4. **异常-回滚联动**：`03-runtime-behavior.md` §3.4 中标记"触发回滚"的类别是否在 `05-ops-governance.md` §2 中有对应步骤
5. **性能-部署匹配**：`04-quality-attributes.md` §2 的 QPS 预估与 §5 的部署拓扑节点数/规格是否匹配
6. **安全-接口契约一致性**：`04-quality-attributes.md` §1 的安全方案要求的认证方式是否与 `02-data-flow.md` §2 的通信模式兼容
7. **ADR 溯源**：`01-architecture-core.md` §2.3/§4 的每个 ADR 是否能在 `competitive-analysis.md` 中找到支撑

评分标准：
- ❌ 阻断项 ≥ 1 → 🔴 BLOCKER
- ⚠️ 警告项 ≥ 3 → 🟡 WARNING
- 全部 ✅ 通过 → 允许进入 Gate 2

#### UAT 报告质量检查（UAT 阶段，V2.1 新增）

检查 `uat-report.md` 的完整性与质量：
1. **用户故事覆盖度**：`uat-report.md` 中验证的用户故事是否覆盖所有 P0 需求（追溯至 `02-requirements-list.md`）
2. **测试环境信息**：是否明确记录测试环境版本、数据库状态、第三方服务可用性
3. **结果分类**：每条验证是否明确标记为 通过 / 不通过 / 有条件通过 / 未测试
4. **缺陷记录完整性**：不通过项是否包含：现象描述、复现步骤、截图/日志、严重级别、指派负责人
5. **遗留问题清单**：是否明确列出遗留问题及下一迭代处理计划
6. **与 code-review-report 的一致性**：UAT 发现的代码缺陷是否与 code-review 中的问题不重复（若重复，标记为流程效率问题）

评分标准：
- P0 用户故事未验证 ≥ 1 个 → 🔴 BLOCKER
- 缺陷记录缺少复现步骤或严重级别 → 🟡 WARNING
- 遗留问题无处理计划 → 🟡 WARNING

#### 详细设计文档质量检查（阶段 4，V2.3 新增）

检查 `feature-*/` 下 5 个设计文件的规范性与可编码性：
1. **规格充分性判定（借鉴 reviewing-design-docs）**：对 `api-spec.md` 和 `db-schema.md` 中的每个规格项判定 SPECIFIED / VAGUE / MISSING。检测模糊语言："标准方案"、"按需"、"TBD"、"as needed"。
2. **数据库与数据架构一致性**：`db-schema.md` 的表清单、实体关系是否与概要设计 `02-data-flow.md` §1 一致；字段类型是否与技术选型匹配。
3. **API 与接口契约一致性**：`api-spec.md` 的通信模式、版本策略是否与概要设计 `02-data-flow.md` §2 一致；URI 设计是否资源导向（禁止动词 URI）。
4. **状态机兼容性**：`state-machine.md` 的局部状态是否与 `03-runtime-behavior.md` §1 的全局状态兼容，无冲突转换。
5. **类设计覆盖功能点**：`design.md` 是否覆盖 `spec.md` 中的所有功能点与验收标准（AC）。
6. **测试计划追溯完整性**：`test-plan.md` 的每个测试用例是否至少追溯到一个 AC；Given/When/Then 格式是否规范。
7. **模块间接口契约审计（借鉴 reviewing-impl-plans）**：当存在 2+ 模块时，检查模块间接口的 request/response/error 格式是否显式定义，数据类型是否兼容。

评分标准：
- 规格项 MISSING ≥ 1 或 VAGUE ≥ 5 → 🔴 BLOCKER
- 状态机与全局状态机冲突 → 🔴 BLOCKER
- 功能点未覆盖 ≥ 1 → 🔴 BLOCKER
- 模块间接口数据类型不兼容 → 🔴 BLOCKER
- VAGUE 数量 3-4 → 🟡 WARNING
- 其他单项缺失 → 🟡 WARNING

#### 文档格式合规性检查（DocForge 规范，V3.1 新增）

检查所有过程文档是否符合 DocForge 文档规范（YAML Front Matter + 锚点ID + C4 绑定）：

1. **YAML Front Matter 存在性与可解析性**：
   - 每个 Markdown 文件首行是否为 `---`？
   - YAML 内容是否可被标准解析器解析（无语法错误）？

2. **字段合法性**：
   - `doc_type` 是否在合法枚举中（PRD/ARCH/DETAIL_DESIGN/API_DESIGN/DB_DESIGN/...）？
   - `fragment_id` 是否符合 `{type}-{iteration}-{seq}` 格式？
   - `version` 是否为有效 SemVer（`MAJOR.MINOR.PATCH`）？
   - `status` 是否为 `DRAFT` / `REVIEW` / `FROZEN` / `DEPRECATED` 之一？
   - `version_type` 是否为 `BASELINE` 或 `DELTA`（当前阶段固定为 BASELINE）？

3. **C4 绑定层级一致性**：
   - `PRD` → `c4_binding.level` 必须为 `"L1"`
   - `ARCH` / `DB_DESIGN` → `c4_binding.level` 必须为 `"L2"`
   - `DETAIL_DESIGN` / `API_DESIGN` → `c4_binding.level` 必须为 `"L3"`
   - 架构类文档是否缺失 `c4_binding` 整体块？

4. **锚点ID 规范性**：
   - 所有 `##` / `###` 标题是否附加 `{#sec-xxx}` 锚点？
   - 锚点ID 是否只含小写字母、数字、下划线？
   - 同文档内锚点ID 是否有重复？
   - `#` 一级标题（文档标题）是否未加锚点（应由 YAML `title` 覆盖）？

5. **C4 标签映射表与 c4_binding 双向一致性（V3.2 修订）**：
   - 文档末尾是否存在《C4 标签映射表》？
   - `c4_binding.actors` 中每个 actor，映射表中是否至少有一条对应记录？
   - `c4_binding.external_systems` 中每个系统，映射表中是否至少有一条对应记录？
   - `c4_binding.containers` 中每个容器，映射表中是否至少有一条对应记录？
   - `c4_binding.components` 中每个组件，映射表中是否至少有一条对应记录？
   - `c4_binding.interfaces` 中每个接口，映射表中是否至少有一条对应记录？
   - 映射表中出现的每个架构元素，其 ID 是否在 `c4_binding` 中有对应定义？

6. **依赖声明完整性**：
   - `dependencies` 数组是否存在且格式正确（每项含 `fragment_id` + `version`）？
   - `dependencies` 中引用的上游文档是否存在于磁盘？
   - `dependencies` 中的版本号是否与上游文档当前 `version` 一致？

评分标准：
- YAML Front Matter 无法解析 → 🔴 BLOCKER
- `doc_type` 与 `c4_binding.level` 不匹配 → 🔴 BLOCKER
- 架构类文档缺失 `c4_binding` → 🔴 BLOCKER
- `c4_binding` 中定义的架构元素在《C4 标签映射表》中无对应记录 → 🟡 WARNING
- 《C4 标签映射表》中引用的 ID 在 `c4_binding` 中不存在 → 🟡 WARNING
- `dependencies` 版本与上游不一致 → 🟡 WARNING
- 锚点ID 缺失 > 2 处 → 🟡 WARNING
- 锚点ID 格式错误或重复 → 🟡 WARNING

### Step 3: 结果聚合与门控判断（三层检查点映射）

`self-check` 作为 Skill 执行规范框架中的 **Gate Out** 标准收口，其判定逻辑映射如下：

| 本框架检查点 | `self-check` 对应位置 | 判定标准 |
|-------------|---------------------|---------|
| **Gate In** | 各 skill 自行实现（输入合法性） | 输入 schema 校验、上下文完整性、权限检查 |
| **Gate Mid** | 各 skill 自行实现（规划可执行性） | 任务 DAG 无环、资源预算可接受、范围明确 |
| **Gate Out** | `self-check` Step 3（本步骤） | **内容一致性 + 内容完整性 + 无内部矛盾** 均通过 |

> **分工说明**：`self-check` 专注于 Gate Out（输出完整性检查）。Gate In 和 Gate Mid 由上游生成 skill（如 `brainstorming`、`prd-generation`、`writing-plans`）在各自执行流程中自行实现。

严重级别分类：
- 🔴 **BLOCKER**：必须修复，否则禁止进入下一阶段（对应 Gate Out 失败）
- 🟡 **WARNING**：建议修复，可进入下一阶段但需记录风险（对应 Gate Out 有条件通过）
- 🟢 **INFO**：优化建议，不影响流程

门控规则（Gate Out）：
- 存在 BLOCKER → **Gate Out 失败**，阶段状态锁定，要求修复后重新执行 self-check
- 无 BLOCKER，存在 WARNING → **Gate Out 有条件通过**，允许进入下一阶段，但记录风险
- 全部通过 → **Gate Out 通过**，阶段状态更新为"已自查通过"

## 与 doc-quality-gate 的关系（分工边界）

`self-check` 与 `doc-quality-gate` 是互补关系，**不可相互替代**：

| 维度 | `self-check` | `doc-quality-gate` |
|------|-------------|-------------------|
| 职责 | **阶段门控**：全生命周期产出物审查 | **文档修复**：多文档一致性检查与自动修复 |
| 行为 | **只读**，不修改任何文件 | **可自动修复**数值/术语/状态/格式问题 |
| 范围 | 文档 + 代码 + 测试 + UAT + 发布 | 仅文档（Markdown / YAML） |
| 输出 | Markdown 自查报告 | Markdown 报告 + YAML 修复日志 |

**协作流程：**
1. 阶段完成后，**先执行 `self-check`** 做广度门控
2. 若 `self-check` 发现文档类 BLOCKER（数值矛盾、术语不一致、状态标记混乱、引用失效），**建议用户调用 `doc-quality-gate`** 执行深度检查与自动修复
3. `doc-quality-gate` 完成自动修复后，**必须重新执行 `self-check`** 确认该阶段通过，方可进入下一阶段

**禁止行为：**
- `self-check` **不得**自动修改产出物（包括调用 `doc-quality-gate` 的修复逻辑）
- 用户**不得**用 `doc-quality-gate` 替代 `self-check` 作为阶段切换的门控依据（`doc-quality-gate` 不检查代码、测试、UAT 等非文档内容）

## Gotchas

- **不自查不流转**：self-check 是阶段切换的**必要条件**，未通过 BLOCKER 时严禁进入下一阶段
- **修复后必须重查**：发现问题修复后，必须重新执行 self-check 确认问题解决，禁止凭感觉跳过
- **上游文档缺失**：若上游依赖文档不存在，标记为 BLOCKER 并要求先补全上游
- **大文档防溢出**：子检查只读取相关章节，不加载全文；必要时分段扫描
- **误报控制**：引入"需澄清"中间状态，不确定时标注为 WARNING 而非 BLOCKER，避免过度拦截
- **与 progress-tracker 联动**：self-check 通过后，progress-tracker 才更新阶段状态；self-check 只输出报告，不直接修改进度文件
- **测试有效性优先于覆盖率**：覆盖率 100% 但全是 Green Mirage（虚假信心）仍然视为 WARNING
- **不要自动修复**：self-check 只报告问题，不自动修改产出物；所有修复需经用户确认后执行
- **交互规格检查不可跳过**：对强交互产品（如 reelforge），按钮级状态机遗漏是用户体验不一致的主因，必须严格检查
- **UAT 报告必须可追溯**：uat-report.md 中的每条验证必须能追溯到上游用户故事（US-XXX），无追溯的验证视为无效
- **发布前必须检查 rollback-plan 时效性**：若代码版本已更新但 rollback-plan 仍指向旧版本，标记为 BLOCKER
