# Architecture Review Guide（架构审查指南）

> 架构设计审查专项指南。供 `code-reviewer` Skill 在 Phase 2 按需加载注入。
> 触发条件：新增模块、接口变更、重大重构、或用户明确请求架构审查。

## SOLID 原则检查

- [ ] **单一职责（SRP）**：每个类/模块是否只有一个变化理由？
- [ ] **开闭原则（OCP）**：扩展行为是否通过新增代码而非修改现有代码？
- [ ] **里氏替换（LSP）**：子类是否能完全替代父类而不破坏行为？
- [ ] **接口隔离（ISP）**：接口是否小而专注？（避免胖接口）
- [ ] **依赖倒置（DIP）**：高层模块是否依赖抽象而非具体实现？

## 耦合与内聚

- [ ] 模块间耦合是否最小？（降低变更传播范围）
- [ ] 是否使用了依赖注入而非硬编码依赖？
- [ ] 是否存在循环依赖？（A → B → C → A）
- [ ] 通用逻辑是否被提取到共享层？
- [ ] 领域逻辑是否泄漏到表现层/基础设施层？

## 抽象与泛化

- [ ] **YAGNI**：未被调用的"完善实现"是否建议删除？
- [ ] 泛化是否等到第三次使用场景才引入？
- [ ] 抽象是否"赚了"它的复杂度？（避免过度工程）
- [ ] 接口设计是否稳定？（避免频繁变更导致大面积修改）

## 数据流与状态管理

- [ ] 数据流是否单向可追踪？（避免隐式状态变更）
- [ ] 全局状态是否最小化？
- [ ] 状态是否放在最合适的层级？（避免 prop drilling 或过度提升）
- [ ] 异步数据流是否有明确的加载/错误/成功状态？
- [ ] 事件驱动架构中，事件契约是否稳定且版本化？

## 扩展性与弹性

- [ ] 水平扩展是否无状态？（会话、本地缓存外置）
- [ ] 关键路径是否有降级策略？（熔断、限流、兜底数据）
- [ ] 配置是否外部化？（环境变量 > 配置文件 > 硬编码）
- [ ] 新模块的启动/停止是否独立？（微服务/模块化 monolith）

## 技术债务识别

- [ ] 是否引入了临时方案（hack）而没有 TODO/FIXME 标记？
- [ ] 是否复制了现有代码而非复用？（DRY 原则）
- [ ] 是否有技术债的偿还计划？（记录在 decisions.md）
- [ ] 旧模式迁移是否 halfway？（新旧模式并存导致混乱）

## 分层架构检查

### 典型分层
```
Presentation（表现层）
  ↓ 依赖
Application（应用层 / Use Case）
  ↓ 依赖
Domain（领域层 / 业务逻辑）
  ↓ 依赖
Infrastructure（基础设施层 / DB、HTTP、文件系统）
```

- [ ] 依赖方向是否始终向下？（上层依赖下层，禁止反向）
- [ ] 领域层是否独立于框架？（可在不修改领域代码的情况下替换框架）
- [ ] 基础设施代码是否通过接口注入？

## API 与接口设计

- [ ] 接口契约是否稳定？（字段增删是否兼容）
- [ ] 错误响应格式是否一致？（状态码、错误码、可读消息）
- [ ] 版本控制策略是否明确？（URL / Header / 媒体类型）
- [ ] 幂等性：重试是否安全？（POST 是否有幂等键）
- [ ] 批量接口是否限制大小？（避免单请求拖垮系统）

## 审查输出标记

- 🔴 **blocking**：循环依赖、严重违反 SRP、领域逻辑泄漏到基础设施层
- 🟠 **important**：缺少抽象、过度工程、API 契约不稳定、缺少降级策略
- 🔵 **suggestion**：引入依赖注入、优化分层、提取共享逻辑
- 📚 **learning**：对复杂架构决策的教育性说明
