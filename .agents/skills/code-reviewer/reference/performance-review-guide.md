# Performance Review Guide（性能审查指南）

> 全语言通用性能审查清单。供 `code-reviewer` Skill 在 Phase 2 按需加载注入。
> 触发条件：变更 >200 行、涉及数据库/缓存/网络、或用户明确请求性能审查。

## 数据库性能

- [ ] 是否存在 N+1 查询？（循环中逐条查询关联数据）
- [ ] 列表查询是否缺少分页或限制？（LIMIT/OFFSET、cursor pagination）
- [ ] 是否缺少必要的索引？（WHERE、JOIN、ORDER BY 字段）
- [ ] 大表查询是否全表扫描？（EXPLAIN 分析）
- [ ] 是否使用了 SELECT * 而非只查询需要的字段？
- [ ] 批量操作是否使用批量 INSERT/UPDATE 而非逐条？
- [ ] 事务范围是否最小化？（避免在事务中执行非 DB 操作）
- [ ] 是否滥用 ORM 导致生成低效 SQL？

## 算法与复杂度

- [ ] 嵌套循环是否会导致 O(n²) 或更高复杂度处理大数据？
- [ ] 递归是否有深度限制？（避免栈溢出）
- [ ] 排序是否必要？（能否用哈希表替代 O(n log n)）
- [ ] 字符串拼接是否在循环中使用？（应使用 StringBuilder/join）
- [ ] 正则表达式是否高效？（避免回溯灾难）

## 内存与资源

- [ ] 大数据集是否使用流式/分页处理而非一次性加载到内存？
- [ ] 是否持有大量对象引用导致无法 GC？
- [ ] 缓存策略是否合理？（TTL、最大容量、淘汰策略）
- [ ] 连接池（DB/HTTP/Redis）是否配置合理？
- [ ] 文件句柄、网络连接、锁是否正确释放？（try-finally / defer / with）
- [ ] 是否有 goroutine / 线程泄漏？（无限制创建且未回收）

## 并发与异步

- [ ] 同步 I/O 是否阻塞了主线程/事件循环？
- [ ] 并发控制是否缺失？（无界 goroutine / 线程池）
- [ ] 锁粒度是否过大？（避免全局锁）
- [ ] 是否存在竞态条件？（读写共享状态无同步）
- [ ] 异步任务失败是否有重试和降级策略？
- [ ] 是否使用了适当的并发原语？（channel vs mutex vs atomic）

## 前端/客户端性能

- [ ] 是否减少了不必要的重渲染？（React: memo/useMemo；Vue: computed）
- [ ] 大列表是否使用虚拟滚动？
- [ ] 图片/资源是否懒加载？
- [ ] 是否避免了大型依赖打包？（tree-shaking、动态导入）
- [ ] API 调用是否可合并或缓存？
- [ ] 是否有阻塞主线程的耗时计算？（Web Worker / 分片）

## 网络与 I/O

- [ ] API 响应是否可缓存？（Cache-Control、ETag）
- [ ] 是否重复请求相同数据？（客户端缓存、dedupe）
- [ ] 文件上传是否分片？大文件是否走直传？
- [ ] 是否压缩了传输内容？（gzip/brotli）
- [ ] DNS/连接/SSL 是否有连接复用？（HTTP keep-alive、连接池）

## 基础设施与部署

- [ ] 是否有冷启动问题？（Serverless 函数初始化时间）
- [ ] 健康检查端点是否轻量？（避免执行重逻辑）
- [ ] 日志输出是否过于频繁？（异步日志、采样）
- [ ] 监控指标采集是否有开销？（避免同步上报阻塞请求）

## 量化目标参考

| 指标 | 目标值 | 阻塞阈值 |
|------|--------|---------|
| API P99 延迟 | < 200ms | > 1s |
| 页面 FCP | < 1.8s | > 3s |
| DB 查询（单条） | < 50ms | > 500ms |
| 内存使用增长 | 稳定 | 持续泄漏 |
| 并发连接数 | < 池上限 | 池耗尽 |

## 审查输出标记

- 🔴 **blocking**：N+1 查询导致严重延迟、无界数据加载可能导致 OOM、死锁
- 🟠 **important**：缺少分页、大事务、明显复杂度问题
- 🔵 **suggestion**：增加缓存、优化索引、使用批量操作
