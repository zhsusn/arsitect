# Code Quality Universal（通用质量反模式）

> 跨语言通用代码质量反模式清单。供 `code-reviewer` Skill 在 Phase 3 按需加载注入。
> 默认加载，除非审查范围极小或用户明确排除。

## 复用审计（Acceptance Checklist）

在批准新代码前，搜索现有代码库：
- [ ] 是否已有工具函数/辅助类可完成同样功能？
- [ ] 是否已有第三方库/标准库函数可替代？
- [ ] 如果功能相似但不完全相同，是否应提取公共部分？

## 参数膨胀（Parameter Sprawl）

- [ ] 函数参数数量是否 > 5？
- [ ] 是否有一组参数总是在多个函数中一起传递？（应封装为对象/上下文）
- [ ] 布尔参数是否用于切换函数行为？（应考虑拆分为两个函数）

## 抽象泄漏（Leaky Abstraction）

- [ ] 抽象层是否暴露了底层实现细节？
- [ ] 调用方是否需要知道内部数据结构才能正确使用？
- [ ] 异常类型是否暴露了底层技术（如 SQL 异常穿透到领域层）？

## 嵌套条件（Nested Conditionals）

- [ ] 嵌套深度是否 > 3？
- [ ] 是否可通过卫语句（guard clause）提前返回简化？
- [ ] 是否可通过多态/策略模式消除条件分支？

## 字符串类型化（String Typing）

- [ ] 是否用字符串传递本应强类型的数据？（如状态、类型标签、ID）
- [ ] 字符串拼接是否用于结构化数据？（应使用对象/结构体/枚举）
- [ ] 魔法字符串是否提取为常量/枚举？

## TOCTOU（Time-of-Check to Time-of-Use）

- [ ] 检查时与使用时之间，状态是否可能被其他线程/进程修改？
- [ ] 文件是否存在检查后、打开前被替换？
- [ ] 权限检查后在执行前是否可能过期？

## 空操作更新（No-Op Update）

- [ ] 数据库 UPDATE 是否可能未修改任何字段？
- [ ] 是否比较了新旧值以避免无意义写入？
- [ ] 批量更新后是否检查了 affected rows？

## 冗余状态（Derived State）

- [ ] 状态是否可从其他状态计算得出？（应使用计算属性/派生）
- [ ] 同步更新的多个状态是否可能不一致？
- [ ] 是否缓存了本可实时计算的值而未处理失效？

## 魔法数字与魔法字符串

- [ ] 数值常量是否有命名？（超时 30 → TIMEOUT_SECONDS）
- [ ] 字符串标识符是否集中管理？
- [ ] 配置值是否硬编码在业务逻辑中？

## 注释质量

- [ ] 注释是否解释了"为什么"而非"做了什么"？
- [ ] 过时注释是否被更新？（代码改了，注释未改比无注释更糟）
- [ ] 是否用注释掩盖了本应通过命名自解释的代码？
- [ ] TODO/FIXME 是否有对应任务 ID 或截止日期？

## 错误处理

- [ ] 异常是否被吞掉？（空 catch / except: pass）
- [ ] 错误信息是否对调用方有用？（"出错了" → "用户 ID 格式无效，期望 UUID"）
- [ ] 是否区分了可恢复错误与不可恢复错误？
- [ ] 失败操作是否有补偿或回滚机制？

## 测试相关反模式

- [ ] 测试是否依赖实现细节而非行为？（重构导致测试失败 = 测试有问题）
- [ ] 测试数据是否使用了共享可变状态？
- [ ] 单元测试是否覆盖了边界情况？（空、极大值、特殊字符）
- [ ] 测试名称是否描述了行为？（`test_user_creation` → `test_creates_user_with_valid_email`）
- [ ] 是否有测试只验证了 mock 被调用？（这种测试没有价值）

## 审查输出标记

- 🟠 **important**：明显可提取的重复代码、参数膨胀、严重嵌套、TOCTOU 风险
- 🟡 **nit**：魔法数字、命名不规范、注释风格
- 📚 **learning**：对复杂模式的教育性说明
- 🌟 **praise**：复用现有抽象、优雅的错误处理、清晰的命名
