## 1. 设计概述

### 1.1 设计目标

基于 PRD-000 v2.0-patch2 与 HLD-000~HLD-005 设计文档，以及实际用户心流（立项 → 脑暴 → 需求确认 → 设计 → 任务拆解 → 编码/测试/Bug 修复），重新组织前端页面导航与功能聚集方式，使：

1. **项目中心**仅负责项目元数据管理，不触发任何 Skill 执行；
2. **需求设计室**成为需求分析、架构设计、设计产物治理的沉浸式工作空间；
3. **开发执行**聚焦 Build/Verify 阶段的任务拆解、编码、测试与问题反馈；
4. **架构治理**前置到设计阶段，作为设计产物基线化与变更影响分析的核心能力；
5. **复杂度评估**不再作为独立页面，其产物（规模初估）作为概要需求阶段的参考数据展示。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **阶段聚集** | 同一 SDLC 阶段的操作、产物、审查、治理功能聚集在同一页面或相邻导航节点 |
| **触发收敛** | Skill 执行触发入口按阶段收敛，避免在拓扑图与详情面板间反复横跳 |
| **迭代闭环** | 需求设计室内部完成"执行 → 审查 → 批注 → 重新生成 → 版本对比"的内循环 |
| **反馈通路** | 开发执行中的问题可通过"执行问题"节点直接反馈回需求设计室，触发设计变更 |
| **基线前置** | 架构治理（基线化、Stale 传播）发生在设计阶段，而非开发完成后 |

### 1.3 术语与引用

| 术语 | 定义 | 来源文档 |
|------|------|----------|
| **需求设计室** | 本设计新增的核心页面，覆盖 Align/Contract 阶段的需求分析与架构设计 | 本设计 |
| **任务中心** | 开发执行的核心页面，负责任务拆解、执行、Bug 修复 | 本设计 |
| **执行问题** | 开发执行中产生的异常记录，支持反馈回架构 | 本设计 |
| **设计产物** | 需求设计室内部产物总览，含 PRD、HLD、DD、C4 DSL、接口契约等 | 本设计 |
| **架构治理** | 设计产物基线化、变更影响分析、Stale 传播控制 | HLD-002 §1.3 |
| **PocketFlow** | Skill 执行三阶段生命周期：prep → exec → post | PRD-000 §2.1 |
| **Gate** | 人工审批节点，阻塞下游 Stage 解锁 | PRD-000 §5.2 |

## 3. 项目中心

### 3.1 设计约束

**项目中心为纯管理域，禁止出现任何 Skill 执行触发入口。** 所有触发执行的操作必须引导用户进入需求设计室或开发执行。

### 3.2 项目工作台

#### 3.2.1 功能清单

| 功能点 | 优先级 | 说明 | 来源需求 |
|--------|--------|------|---------|
| 项目 CRUD | P0 | 创建/编辑/删除项目，项目名称 1-100 字符，唯一性校验 | REQ-P0-001 |
| Application 绑定 | P0 | 项目必须绑定一个 Application，创建时引导选择 | §2.1 四层模型 |
| Draft/Active 双态流转 | P0 | Draft 项目仅允许分析型 Skill；Active 项目允许全阶段 Skill；流转需人工确认 | REQ-P0-014 |
| 项目列表 | P0 | 搜索、排序、状态筛选（Draft/Active/Archived/Cancelled） | REQ-P0-002 |
| 健康度卡片 | P0 | 展示进度、风险、待审批 Gate 数量 | REQ-P0-022 |
| 风险预警 | P0 | 超时预警、Stale 标记、阻塞节点提示 | REQ-P0-023 |
| 7 天自动清理 | P0 | Draft 项目 7 天无活动自动归档为 Cancelled（BR-013） | BR-013 |
| 快速进入设计室 | P0 | 项目卡片上显式展示"进入需求设计室"主按钮，引导用户下一步 | 本设计 |

#### 3.2.2 页面布局

```
┌─────────────────────────────────────────────┐
│ 顶部：项目中心 / 项目工作台                    │
├─────────────────────────────────────────────┤
│ 操作栏：[新建项目] [导入项目] [筛选 ▼] [搜索]  │
├─────────────────────────────────────────────┤
│ 统计卡片：活跃项目 3 | 待审批 Gate 2 | 风险 1 │
├─────────────────────────────────────────────┤
│ 项目列表（卡片/表格双视图）                    │
│ ┌─────────────┐ ┌─────────────┐              │
│ │ 项目名称     │ │ 项目名称     │              │
│ │ App: 订单系统 │ │ App: 支付系统 │              │
│ │ 状态: Draft  │ │ 状态: Active │              │
│ │ 健康度: 正常 │ │ 健康度: 风险 │              │
│ │ [进入需求设计室]│ │ [进入开发执行] │              │
│ └─────────────┘ └─────────────┘              │
└─────────────────────────────────────────────┘
```

**关键交互**：
- 点击"新建项目" → 弹窗表单（项目名称、描述、选择 Application）→ 创建后状态 = Draft → 卡片显示"进入需求设计室"按钮
- 点击"进入需求设计室" → 路由跳转至 `/requirement-studio/概要需求?projectId=xxx`
- 点击"进入开发执行" → 路由跳转至 `/execution/任务中心?projectId=xxx`（仅 Active 项目显示）

### 3.3 应用管理

#### 3.3.1 功能清单

| 功能点 | 优先级 | 说明 |
|--------|--------|------|
| 应用 CRUD | P0 | 应用名称、技术栈标签、描述 |
| 技术栈锁定 | P0 | 绑定技术栈（如 React/FastAPI/SQLite），影响后续模板推荐 |
| 历史迭代继承 | P1 | 查看该应用下历史项目的产物继承关系 |
| 模块清单管理 | P1 | 预定义应用内模块（如用户系统、订单系统） |

#### 3.3.2 与项目创建的关系

创建项目时，若无可选 Application，系统引导用户先进入"应用管理"创建 Application。Application 创建后才允许创建项目。

## 4. 需求设计室

### 4.1 总体设计

需求设计室是**沉浸式工作空间**，采用**顶部阶段导航条 + 三栏布局**（左侧任务树 / 中间产物 / 右侧审查与执行）。

#### 4.1.1 进入条件

- 项目状态 = Draft 或 Active
- 若项目为 Draft，仅"概要需求"阶段可用，其余阶段置灰提示"需先完成立项"

#### 4.1.2 全局布局

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 顶部阶段导航：[概要需求] → [详细需求] → [概要设计] → [详细设计] → [设计产物] → [架构治理] │
│              当前高亮：概要需求                                            │
├──────────┬──────────────────────────────┬──────────────────────────────┤
│          │                              │                              │
│ 左侧：    │      中间：产物工作区          │     右侧：审查与执行控制台    │
│ 阶段任务树 │                              │                              │
│          │  ┌────────────────────────┐    │  ┌────────────────────────┐  │
│  ▼ 概要需求│  │ Tab: PRD / 用户故事 /  │    │  审查批注面板            │  │
```

### 4.2 概要需求

#### 4.2.1 阶段目标

完成需求对齐（Align），产出：脑暴纪要继承、用户故事、PRD（概要）、草图（PageSpec）、验收标准初稿。

#### 4.2.2 左侧任务树

| 任务节点 | 状态 | 操作 |
|----------|------|------|
| 脑暴纪要 | 继承自 Draft | 查看，不可编辑（Draft 阶段产物） |
| 用户故事 | NOT_STARTED / REVIEW_PENDING | [执行 Skill] / [重新生成] |
| PRD（概要）| NOT_STARTED / REVIEW_PENDING | [执行 Skill] / [重新生成] |
| 草图 | NOT_STARTED / REVIEW_PENDING | [执行 Skill] / [重新生成] |
| 验收标准 | NOT_STARTED / REVIEW_PENDING | [执行 Skill] / [重新生成] |

**触发方式**：点击任务节点 → 中间产物区切换 → 右侧执行面板显示对应 Skill 指令快照 → 点击"执行"或"重新生成"。

#### 4.2.3 草图（PageSpec）渲染

- 基于用户故事自动生成低保真页面逻辑图（文本框+箭头，标注字段、按钮、跳转）
- 字段覆盖率 ≥90%（REQ-P0-040）
- 缺失字段检测：若用户故事未描述字段，草图区域显示"[未定义字段]"占位并标红

#### 4.2.4 复杂度评估（下沉）

不再作为独立页面。在概要需求阶段底部展示**规模初估参考卡片**：

```
┌─────────────────────────────┐
│ 规模初估参考（基于脑暴结果）  │
│ 模块数: 3 | 接口数: 8 | 页面数: 5 │
│ 技术复杂度: 中 | 风险等级: 低      │
│ 推荐路径: Standard（参考）      │
│ [进入模板配置查看详情]          │
└─────────────────────────────┘
```

此卡片仅作参考，不阻塞流程，最终路径由用户在平台管理-模板配置中预设或手动选择。

### 4.3 详细需求

#### 4.3.1 阶段目标

详细化需求，产出：详细 PRD、验收标准、接口契约初稿（YAML）。

#### 4.3.2 解锁条件

概要需求阶段全部任务通过"需求确认 Gate"（或用户确认进入详细需求，单人场景下可简化）。

#### 4.3.3 接口契约初稿

- 产物格式：OpenAPI 3.0 YAML
- 渲染方式：Swagger UI 嵌入式预览（中间产物区）
- 编辑模式：YAML 文本编辑器，保存时触发 schema 校验

### 4.4 概要设计

#### 4.4.1 阶段目标

完成概要设计（HLD），产出：HLD 文档、C4 L1/L2 DSL、线框图（初步布局）。

#### 4.4.2 C4 架构（L1/L2）

- 自动生成：基于 HLD 文档解析生成 `c4-l1.dsl.yml`、`c4-l2.dsl.yml`
- 渲染：Mermaid.js 浏览器渲染 Context/Container 图
- 手动覆盖：支持 DSL 文本编辑，保存后标记 `manual_override=true`，后续不再自动覆盖
- 层级穿透：点击 Context 边界 → 下钻至 Container 图，面包屑同步更新

#### 4.4.3 线框图

- 基于 C4 Container 图初步映射页面结构
- 产出：SVG 线框图（页面容器、导航栏、内容区占位）
- 与详细设计的 OpenUI 区分：线框图为低保真结构草图，OpenUI 为高保真可交互原型

### 4.5 详细设计

#### 4.5.1 阶段目标

完成详细设计（DD），产出：DD 文档、C4 L3/L4 DSL、OpenUI 原型、数据绑定、DB 设计。

#### 4.5.2 C4 架构（L3/L4）

- 自动生成：基于 DD 文档和接口契约生成 Component/Code 级 DSL
- 反向代码定位：Component 节点绑定 `arsitect.codegen_target` 字段，点击可打开本地 IDE 对应文件（P0 需求 REQ-P0-033）
- 准确率不足时降级：自动生成准确率 < 60% 时，仅展示 L1/L2，L3/L4 提示"需手动补充"

#### 4.5.3 OpenUI 原型

- 触发方式：C4 Container 图 + 接口契约 → 生成 OpenUI 提示词 → 调用 Docker HTTP API
- 渲染：中间产物区嵌入 iframe 展示可交互 HTML
- 降级：OpenUI 服务不可用时，自动降级为 Wireframe 静态预览（ADR-003）

#### 4.5.4 数据绑定

- 功能：检测原型页面与 C4 接口的覆盖度
- 操作：点击"检查覆盖度" → 系统对比页面元素与接口契约 → 缺失接口标红
- 回写：点击"一键回写 C4 DSL" → 自动更新 `arsitect.aac.yml` 并标记变更待评审

#### 4.5.5 DB 设计

#### 4.7.4 与开发执行的联动

- **基线化后**：开发执行任务中心读取基线产物作为输入，任务拆解基于基线版本进行
- **Stale 传播**：若基线产物在开发执行阶段被修改（如通过"执行问题"反馈回架构），系统自动：
  1. 标记产物为 Stale
  2. 计算受影响下游任务（开发执行任务中心中的任务）
  3. 推送通知到开发执行-执行问题
  4. 可选：自动创建变更请求（CR）

## 5. 开发执行

### 5.1 总体设计

开发执行聚焦 **Build（编码）** 与 **Verify（测试）** 阶段，核心页面为**任务中心**，项目画布作为拓扑总览与快捷入口。

### 5.2 项目画布（Build/Verify 专用）

#### 5.2.1 功能定位

不同于需求设计室的"阶段导航"，项目画布在开发执行中作为**Build/Verify 阶段的拓扑可视化**，仅展示编码、测试、Bug 修复相关节点。

#### 5.2.2 节点类型

| 节点 | 状态 | 操作 |
|------|------|------|
| 编码任务 | NOT_STARTED / IN_PROGRESS / PASSED / BLOCKED | 点击 → 跳转任务中心对应任务 |
| 单元测试 | NOT_STARTED / IN_PROGRESS / PASSED / FAILED | 点击 → 跳转任务中心 |
| 集成测试 | NOT_STARTED / IN_PROGRESS / PASSED / FAILED | 点击 → 跳转任务中心 |
| Bug 修复 | NOT_STARTED / IN_PROGRESS / CLOSED | 点击 → 跳转任务中心 |

**关键约束**：项目画布在开发执行中**不直接触发 Skill**，仅作为导航与状态总览。触发操作在任务中心完成。

### 5.3 任务中心（核心页面）

#### 5.3.1 功能定位

开发执行的**主战场**，覆盖：任务拆解、任务执行（编码/测试）、Bug 修改。

#### 5.3.2 页面布局

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 顶部：开发执行 / 任务中心                                                │
├─────────────────────────────────────────────────────────────────────────┤
│ 阶段切换：[任务拆解] [任务执行] [Bug 修复]                                │
├─────────────────────────────────────────────────────────────────────────┤
│ 左侧：任务/需求树                                                         │
│ 订单模块                                                                  │
│  ├─ 用户管理接口   [编码] [待执行]                                        │
│  ├─ 订单创建接口   [编码] [执行中]  ← 当前选中                            │
│  └─ 订单查询接口   [测试] [待执行]                                        │
│ 支付模块                                                                  │
│  ├─ 支付接口       [编码] [已完成]                                        │
│  └─ 支付回调       [测试] [失败]   ← 标红                                 │
├─────────────────────────────────────────────────────────────────────────┤
│ 中间：任务详情与产物                                                       │

│ 右侧：执行面板                                                            │
│ 触发 Skill: skill-code-generation                                       │

│ [执行编码] [重新生成] [中断执行]                                          │

│ > prep: 加载上下文...                                                     │
│ > exec: 调用 Kimi CLI...                                                │
│ > post: 产物写入...                                                       │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 5.3.3 任务拆解

**触发方式**：
1. 手动：用户点击"新建任务"，选择设计产物作为输入，填写任务描述
2. 自动：基于详细设计产物（HLD/DD/接口契约）一键生成任务清单（调用 Skill 或本地规则）

**任务数据结构**：
```json
{
  "taskId": "T-001",
  "name": "订单创建接口编码",
  "type": "coding", // coding | test | bugfix
  "inputArtifacts": ["api-contract.yaml", "c4-l2.dsl.yml"],
  "status": "NOT_STARTED",
  "assignedSkill": "skill-code-generation",
  "parentModule": "订单模块"
}
```

#### 5.3.4 任务执行

- 选择任务 → 右侧执行面板加载对应 Skill 快照
- 点击"执行编码" → 触发 PocketFlow → 产物写入 `src/` 目录
- 执行完成后状态自动更新，产物浏览器同步

#### 5.3.5 Bug 修复

- 测试失败或执行异常时，任务状态 = FAILED/BLOCKED
- 用户可点击"标记为 Bug" → 创建 Bug 记录 → 进入 Bug 修复 Tab
- Bug 修复支持：重新触发编码 Skill（携带错误日志作为上下文）、或手动修复后上传产物

### 5.4 执行问题

#### 5.4.1 功能定位

记录开发执行中的异常（编译失败、测试未通过、架构实现偏差），并支持**反馈回架构**（触发设计变更）。

#### 5.4.2 问题记录

| 字段 | 说明 |
|------|------|
| 问题类型 | 编译错误 / 测试失败 / 架构偏差 / 接口不匹配 / 其他 |
| 关联任务 | 关联到任务中心的具体任务 |
| 错误日志 | 自动捕获 stdout/stderr 摘要 |
| 关联产物 | 自动关联输入的设计产物（HLD/DD/接口契约） |
| 建议操作 | 重试 / 反馈回架构 / 跳过 |

#### 5.4.3 反馈回架构（关键联动）

操作流程：
1. 用户在执行问题页面查看问题详情
2. 点击"反馈回架构" → 弹窗选择关联设计产物（如 `api-contract.yaml`）
3. 系统：
   - 在需求设计室-架构治理中标记该产物为 Stale
   - 创建变更请求（CR），描述为"接口契约与实际实现偏差"
   - 可选：自动跳转至需求设计室-详细设计，携带错误日志作为参考资料
4. 用户进入需求设计室修改设计 → 重新基线化 → 回到开发执行重新执行任务

### 5.5 执行监控

- 实时展示当前所有 IN_PROGRESS 任务的 PocketFlow 三阶段状态
- SSE 推送日志流，支持多任务并行监控
- 与任务中心联动：点击监控中的任务 → 跳转任务中心详情

### 5.6 AI CLI

- 手动调试入口：输入 Skill 名称、自定义上下文、执行参数
- 主要用于排查执行问题或测试新 Skill
- 输出直接展示在页面，不自动关联项目产物

### 5.7 监控看板

- Token 消耗统计（按项目/阶段/任务）
- 阶段耗时统计（编码阶段平均耗时、测试阶段平均耗时）
- 瓶颈识别：哪个任务类型失败率最高、哪个模块耗时最长

## 9. 核心状态机与数据流

### 9.1 项目级状态机（简化）

```
[*] → Draft : 创建项目
Draft → Active : 用户确认立项（项目工作台操作）
Active → Archived : 全部阶段通关 / 手动归档
Active → Cancelled : 手动取消
Draft → Cancelled : 7 天无活动（BR-013）
```

### 9.2 需求设计室阶段状态机

```
概要需求 → 详细需求 : 概要需求 Gate 通过 / 用户确认
详细需求 → 概要设计 : 详细需求 Gate 通过 / 用户确认
概要设计 → 详细设计 : 概要设计 Gate 通过 / 用户确认
详细设计 → 设计产物 : 设计完成，进入产物整理
设计产物 → 架构治理 : 用户主动基线化
架构治理 → 开发执行 : 基线化完成，进入开发

（任一阶段可退回上一阶段，需确认）
```

### 9.3 任务中心状态机

```
NOT_STARTED → IN_PROGRESS : 用户触发执行
IN_PROGRESS → PASSED : Skill 执行成功，产物校验通过
IN_PROGRESS → FAILED : Skill 执行失败 / 测试未通过
FAILED → IN_PROGRESS : 用户重试（≤3次，BR-005）
FAILED → BUG_OPENED : 用户标记为 Bug
BUG_OPENED → IN_PROGRESS : Bug 修复后重新执行
```

### 9.4 架构治理数据流

```
设计产物（详细设计完成）
    │
    ▼
[创建基线] → 锁定核心产物版本 → 标记 BASELINE
    │
    ▼
开发执行任务中心读取基线产物作为输入
    │
    ▼
执行问题发现偏差 → [反馈回架构]
    │
    ▼
架构治理：标记产物 STALE → 计算影响范围 → 通知任务中心
    │
    ▼
用户选择：回退到基线 / 提交变更请求（CR）/ 直接修改后重新基线化
```

### 11.3 关键接口契约（前后端）

**需求设计室统一入口**：
```
GET /api/v1/requirement-studio/{projectId}/status
→ 返回当前项目各阶段状态（概要需求/详细需求/概要设计/详细设计/基线化）

POST /api/v1/requirement-studio/{projectId}/stage/{stageId}/execute
→ 触发阶段 Skill，body: { skillId, context, referenceMaterials }

POST /api/v1/requirement-studio/{projectId}/stage/{stageId}/review
→ 提交审查，body: { comments, suggestions, references, action: "pass" | "regenerate" }

GET /api/v1/requirement-studio/{projectId}/artifacts
→ 获取设计产物列表（按阶段分组）
```

**任务中心统一入口**：
```
GET /api/v1/execution/{projectId}/tasks
→ 返回任务列表（按模块分组）

POST /api/v1/execution/{projectId}/tasks
→ 创建任务（手动拆解），body: { name, type, inputArtifacts, assignedSkill }

POST /api/v1/execution/{projectId}/tasks/{taskId}/execute
→ 执行任务 Skill

POST /api/v1/execution/{projectId}/issues
→ 创建执行问题，body: { type, taskId, errorLog, suggestedAction }

POST /api/v1/execution/{projectId}/issues/{issueId}/feedback-to-architecture
→ 反馈回架构，body: { targetArtifactId, changeDescription }
→ 后端自动：标记 Stale、创建 CR、通知需求设计室
```

**架构治理统一入口**：
```
POST /api/v1/requirement-studio/{projectId}/governance/baseline
→ 创建基线，body: { artifactIds, description }

GET /api/v1/requirement-studio/{projectId}/governance/stale-analysis
→ 获取 Stale 产物及影响范围

POST /api/v1/requirement-studio/{projectId}/governance/change-request
→ 提交变更请求，body: { targetArtifactId, changeType, reason }
```
